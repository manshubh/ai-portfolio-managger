## Resolved So Far

- Runs are allowed to span multiple days. The workflow does not need to freeze all data to a single-day snapshot.
- Multiple runs in one day are considered rare. If they happen, manual cleanup is acceptable for now.
- `temp/research/` is disposable scratch space. We do not need to archive all intermediate artifacts per run.
- Run entry should begin from a portfolio CSV file. The system should fetch or generate this CSV from Wealthfolio with filtering options, show the fetched output, and only then allow AI analysis to proceed.
- The dumped portfolio CSV becomes the frozen holdings snapshot for that run, even if Wealthfolio changes while the run is still in progress.
- Per-stock thesis should live alongside stock-level data. If it cannot be stored cleanly in Wealthfolio itself, we need a filesystem fallback that is still treated as part of the analysis input.
- A sidecar thesis file keyed by Yahoo ticker is acceptable as the MVP fallback if Wealthfolio cannot store thesis cleanly.
- The per-market thesis sidecar should be the explicit MVP source of truth.
- The normalized run CSV should stay lean. No need to include `sector`, `industry`, or `cash_balance` in the snapshot.
- We should build scripts that read the structured thesis sidecar and write/populate a `thesis` column in the normalized portfolio CSV or equivalent analysis input.
- The normalized snapshot CSV should always include a `thesis` column, even when empty.
- Thesis sidecar storage should be per-market, not one global file.
- The normalized snapshot CSV should include an `asset_type` field so later phases can distinguish `stock` vs `ETF`.
- The normalized snapshot CSV schema is:
  - `ticker`
  - `name`
  - `market`
  - `currency`
  - `account`
  - `account_group`
  - `asset_type`
  - `quantity`
  - `avg_cost`
  - `snapshot_price`
  - `snapshot_market_value`
  - `allocation_pct`
  - `unrealized_pl_pct`
  - `thesis`
- Portfolio-state inputs are frozen for the run once Phase 1 artifacts are dumped.
- Live market data can still be fetched during later analysis phases, but it should flow into per-ticker research files rather than mutating the frozen snapshot CSV.
- Other non-holdings inputs should also be frozen as Phase 1 artifacts where practical, including recent transactions and corporate-actions outputs.
- Reusing `temp/research/` with manual cleanup is acceptable.
- Follow-up on prior recommendations can be inferred from portfolio deltas between runs. Example: if a sell was recommended, the next run checks whether the position was exited or trimmed.
- Drawdown and entry-price comparisons should use average cost, since that is the practical portfolio reference point.
- Confidence can be rubric-based:
  - `HIGH`: strong data completeness, recent sources, no major conflicts, thesis/story clear
  - `MEDIUM`: some missing fields or mixed signals
  - `LOW`: stale or missing data, major source gaps, or unresolved contradictions
- `business_story_status` should remain analyst judgment with guardrails, not a fully deterministic checklist.
- MVP includes both India and US.
- Canonical ticker format should be Yahoo-style symbols.
- Our numbers should be close to Wealthfolio and explainable, not necessarily byte-for-byte identical.
- Parallel research/scoring should be user-configurable. The user should be able to choose batch size / number of stocks per session, while the lock system prevents duplicate work across sessions.
- ETFs should remain in the same snapshot CSV with `asset_type=ETF`, and later phases should use a lighter ETF-specific analysis path.
- ETF analysis should focus on what the ETF tracks, not on wrapper-specific fund details like AUM or expense ratio.
- ETF rows in the final report can keep the same broad template shape as stocks, with conditional analysis content based on `asset_type`.
- `snapshot_price` should come from stored valuation at export time, and the exported snapshot should contain the latest prices available at the moment of export.
- No extra snapshot metadata like `exported_at` or `quotes_as_of` is needed in MVP.

## Proposed Spec Direction

### 1. Account Scope Model

Proposal: every run should accept a scope and produce a reviewable portfolio CSV before research starts.

Suggested model:

- `scope_type = market | account_group | account | consolidated`
- `scope_value = india | us | <group-name> | <account-name> | consolidated`

Behavior:

- `market` aggregates all active accounts for that market.
- `account_group` aggregates all active accounts in the named Wealthfolio group.
- `account` analyzes a single Wealthfolio account in isolation.
- `consolidated` merges completed market runs.

Suggested run flow:

1. User selects scope.
2. System exports a normalized portfolio CSV for that scope.
3. The CSV is dumped to disk as the run snapshot.
4. Later analysis sessions read that dumped snapshot rather than re-fetching holdings.

Notes:

- The system should dump the file; user review is optional, not required.
- There is no mandatory confirmation gate before analysis starts in later sessions.
- This fits the multi-session workflow, where Phase 1 prepares artifacts and later sessions consume them.
- The CSV should be minimal and focused on analysis-relevant portfolio fields.
- Thesis can be merged in by script as a dedicated `thesis` column.
- The snapshot CSV is the frozen portfolio-state input for the run.
- Live prices and other fresh market context can still appear in downstream ticker research files without rewriting the snapshot.

Normalized snapshot CSV columns:

`ticker,name,market,currency,account,account_group,asset_type,quantity,avg_cost,snapshot_price,snapshot_market_value,allocation_pct,unrealized_pl_pct,thesis`

Price freshness note:

- The snapshot must include the latest prices available at export time.
- This likely means either:
  - Wealthfolio export already reflects freshly refreshed quotes, or
  - the workflow must run a recalc/quote refresh step before export.
- This is still a validation item, not yet a confirmed product guarantee.

This keeps the current market-level design intact while supporting cases like multiple India accounts that need separate analysis.

### 2. Business Story Status Rubric

Proposal: keep it analyst-judged, but give the analyst a bounded rubric.

Suggested meanings:

- `intact`
  - Core thesis pillars still hold.
  - No governance red flags.
  - No material capital-allocation or execution breakdown.
  - Weak price action or rich valuation alone cannot downgrade this.

- `weakening`
  - One or two thesis pillars are deteriorating.
  - Execution misses, margin pressure, market-share loss, or management communication issues exist.
  - Concerns are real but not conclusive.
  - This is the main bucket for "watch closely / under review".

- `broken`
  - Governance breakdown, fraud, major regulatory action, thesis invalidation, structural balance-sheet deterioration, or clear loss of the original edge.
  - Can also be triggered by deterioration persisting across multiple runs with no credible recovery evidence.

Possible implementation rule:

- Start at `intact`.
- Move to `weakening` if at least one thesis pillar is materially challenged.
- Move to `broken` only on dealbreakers or repeated weakening with stronger confirming evidence.

Important note:

- This should guide the analyst, not override them mechanically.
- The spec should explicitly allow human-style judgment as long as the rationale cites concrete evidence from the stock file.

### 3. Corporate Actions Handling

Proposal: change from blocking to severity-based.

Suggested model:

- Upcoming events: informational only.
- Recently effective but unrecorded events: high-severity warning.
- Phase 1 continues either way.
- Warnings are written to a dedicated warning file only.

This matches the user's preference that applying actions remains their responsibility.

### 4. Thesis Storage

Proposal: thesis belongs to the stock entity, not to a temporary report input.

Desired priority:

1. MVP source of truth: per-market filesystem sidecar keyed by canonical Yahoo ticker.
2. Wealthfolio-native thesis storage is a future enhancement only if it becomes clearly practical.

Requirements for either approach:

- Missing thesis is valid and treated as empty.
- Thesis must be loaded automatically during analysis, not pasted manually.
- Thesis storage must remain aligned with the rest of the analysis structure across runs.
- The thesis content itself is free-text, but the sidecar container should be structured enough for scripts to read and merge reliably.

Suggested MVP sidecar shape:

```yaml
theses:
  RELIANCE.NS: "Core India energy + retail + telecom compounding thesis..."
  AAPL: "Ecosystem lock-in, services mix shift, buybacks, premium brand durability..."
```

This keeps the payload simple while giving us a deterministic merge format.

Recommended per-market location:

- `input/india/theses.yaml`
- `input/us/theses.yaml`

### 5. ETF Analysis Path

Proposal: ETFs stay in the same snapshot CSV and overall workflow, but their analysis path is different.

Guidance:

- Focus on the underlying index or basket the ETF tracks.
- Analyze the same macro/market/business context that would matter for owning that exposure.
- Do not spend effort on wrapper-specific fund details like AUM, TER/expense ratio, or fund mechanics in MVP unless they become material to the thesis.

Example:

- `NIFTYBEES` should trigger analysis of the Nifty 50 exposure, not a conventional stock-style analysis of the ETF product itself.
- Final report formatting can remain broadly similar to stocks, with ETF-specific conditional content underneath.

## Verified Wealthfolio Notes

Using the official Wealthfolio docs:

- Wealthfolio currently documents both `Account Group` and `Is Active` on the account model.
- `Account Group` is intended to organize accounts.
- `Is Active` controls whether an account is included in portfolio calculations and reporting.
- Wealthfolio officially supports exports to CSV, JSON, and SQL backups.
- Official docs confirm export types like accounts, transactions, and goals.
- I did not find official documentation yet confirming a filtered holdings-snapshot CSV export by account/group.

Sources:

- https://wealthfolio.app/docs/guide/accounts/
- https://wealthfolio.app/docs/guide/settings/
- https://wealthfolio.app/docs/concepts/
- https://wealthfolio.app/docs/guide/data-export/
- https://github.com/afadil/wealthfolio/releases

Implication for our spec:

- We likely do not need a custom tagging system for account selection in MVP.
- We can build run scoping on top of Wealthfolio account names and account groups.
- The "portfolio CSV" entrypoint should not assume Wealthfolio already gives us the exact filtered analysis snapshot we need.
- Safer default: our tooling should be able to generate a normalized CSV from Wealthfolio DB queries after applying scope filters.
- Once generated, that normalized CSV should be treated as the stable run input for all later phases.
- I found official release-note evidence that Wealthfolio has a recalc action that refetches quotes and recalculates portfolio history, but I did not find official documentation proving that export itself triggers quote refresh.
- So the spec should require "latest prices at export time" as an invariant, while leaving open whether that is satisfied by native export or by an explicit pre-export recalc step.
- Exact export implementation details should be researched when this part is implemented.
- Likewise, whether ticker research files should show both `snapshot_price` and `live_price` or only live price is deferred to implementation-time research.
