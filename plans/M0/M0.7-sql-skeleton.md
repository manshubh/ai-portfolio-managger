## Approach

### File header

```sql
-- version: wealthfolio v3.2.1 (build 20260301.1)
-- schema-ref: research/wealthfolio-schema-v3.2.1.txt
-- mode: holdings-only (no reads from the `activities` table)
-- binding: SQLite named parameters (:param)
-- conventions: see "Common idioms" section below
```

A top-of-file banner makes it obvious when future upgrades land — upgrading Wealthfolio is a separate bd ticket per [SPEC §19.14](../../docs/SPEC.md).

### Named-query convention

Each query is introduced by a `-- name: <slug>` marker line. The wrapper (M2) reads the file as text, splits on `^-- name: `, and dispatches by slug. Blocks are separated by a blank line and a `;` terminator.

```sql
-- name: export-snapshot
-- params: :market, :scope_type, :scope_value
-- returns: ticker, name, market, currency, account, account_group,
--          asset_type, quantity, avg_cost, snapshot_price,
--          snapshot_market_value, allocation_pct, unrealized_pl_pct
WITH ...
SELECT ...
;
```

This is intentionally lower-tech than `aiosql` — one regex, no runtime dep, easy to inspect by hand.

### Common idioms (defined once near the top, inlined per query for portability)

**Market mapping** (applied post-join on `assets` + `accounts`):

```sql
CASE
  WHEN a.instrument_exchange_mic IN ('XNSE','XBOM') THEN 'india'
  WHEN a.instrument_exchange_mic IN ('XNAS','XNYS','ARCX') THEN 'us'
  WHEN a.instrument_exchange_mic IS NULL AND acc.currency = 'INR' THEN 'india'
  WHEN a.instrument_exchange_mic IS NULL AND acc.currency = 'USD' THEN 'us'
  ELSE 'unknown'
END AS market
```

**Yahoo-ticker normalization**:

```sql
CASE
  WHEN a.instrument_exchange_mic = 'XNSE' THEN a.instrument_symbol || '.NS'
  WHEN a.instrument_exchange_mic = 'XBOM' THEN a.instrument_symbol || '.BO'
  WHEN a.instrument_exchange_mic IN ('XNAS','XNYS','ARCX') THEN a.instrument_symbol
  ELSE COALESCE(a.display_code, a.instrument_symbol)
END AS ticker
```

**Scope filter** (SPEC §6.8 tri-mode):

```sql
AND (
  (:scope_type = 'market'        AND market           = :scope_value)
  OR (:scope_type = 'account_group' AND acc."group"   = :scope_value)
  OR (:scope_type = 'account'       AND acc.name      = :scope_value)
)
```

`accounts.group` must be quoted — `group` is a reserved word.

**Latest snapshot per account**:

```sql
latest_snapshot AS (
  SELECT hs.*
  FROM holdings_snapshots hs
  JOIN (
    SELECT account_id, MAX(snapshot_date) AS max_date
    FROM holdings_snapshots
    GROUP BY account_id
  ) m ON m.account_id = hs.account_id AND m.max_date = hs.snapshot_date
)
```

### Per-query sketches

#### `export-snapshot` — [SPEC §7.1.1](../../docs/SPEC.md) column contract

Output: the Phase 1 frozen snapshot columns **minus** `thesis` (wrapper-merged). Joins `latest_snapshot` → `json_each(positions)` → `accounts` → `assets` → latest `quotes` (for `snapshot_price`). Computes `snapshot_market_value = quantity * snapshot_price`, `allocation_pct = market_value / SUM(market_value) OVER ()` after scope filter, `unrealized_pl_pct = (snapshot_price - avg_cost) / avg_cost`.

TODOs: `Position` JSON field names (`$.quantity`, `$.avg_cost`) resolved in M2.

#### `list-holdings` — [SPEC §5.3](../../docs/SPEC.md) column contract

Slimmer SELECT over the same base CTEs: `(symbol, currency, quantity, avg_cost, current_price, market_value, allocation_pct)`. No account metadata, no P/L. `current_price` is the latest `quotes.close` (distinct from `snapshot_price` on frozen snapshots, though identical at export time in practice).

#### `get-cash-balance` — [SPEC §6.8](../../docs/SPEC.md)

Params: `:currency`, `:scope_type`, `:scope_value`. v1 reads `daily_account_valuation` (latest row per account), filters to `accounts.currency = :currency` within the scope, `SUM(cash_balance)`. TODO for multi-currency cash balances inside `holdings_snapshots.cash_balances` JSON — M2 extends this once the JSON shape is pinned.

#### `get-net-worth` — [SPEC §5.3](../../docs/SPEC.md)

Params: `:date` (optional; defaults to latest). CTE `valuation_at_date` = latest `daily_account_valuation` row per `account_id` where `valuation_date <= :date`. Returns `SUM(CAST(total_value AS REAL) * CAST(fx_rate_to_base AS REAL))` — values are stored as TEXT so explicit casts matter.

#### `get-avg-cost` — [SPEC §6.8](../../docs/SPEC.md)

Params: `:ticker`. Unrolls `positions` JSON from `latest_snapshot`, applies reverse Yahoo-ticker match on `(instrument_symbol, instrument_exchange_mic)` pairs, returns the weighted average across accounts if multiple hold the same ticker: `SUM(quantity * avg_cost) / SUM(quantity)`.

#### `get-portfolio-twr` — [SPEC §6.8](../../docs/SPEC.md)

Params: `:market`, `:start`, `:end`. v1 returns per-day aggregates — `(date, portfolio_value_base, daily_net_contribution_base)` — summed across accounts in the selected market. The wrapper computes the chained product `PROD(1 + (ΔV - ΔC) / V_prev) - 1`. SQL-side recursive CTE is possible but fragile with TEXT-stored decimals; deferring to Python is cleaner for v1.

### Acceptance of "skeleton" fidelity

A query in this file passes M0.7 review if:

1. Parameter set matches the [SPEC §6.8](../../docs/SPEC.md) sub-command signature.
2. Output columns match SPEC §5.3 / §7.1.1 (or §6.8 for scalar queries).
3. `FROM` / `JOIN` structure references only tables that exist in `research/wealthfolio-schema-v3.2.1.txt`.
4. Scope filter uses the common idiom above.
5. Any unresolved shape (JSON field names, cross-currency cash, TWR chaining) is marked `-- TODO(M2):` inline.

What does **not** need to pass in M0.7: actual execution, ticker normalization edge cases for ETFs / ADRs, currency conversion fidelity beyond `fx_rate_to_base`.

## Risks and open questions

- **R1. `Position` JSON field names may not match the `$.quantity` / `$.avg_cost` placeholders.** Mitigation: explicit `TODO(M2)` markers; M2 verifies against a real snapshot and corrects. Blast radius is 2 queries (`export-snapshot`, `get-avg-cost`), both easy to patch.
- **R2. Yahoo-ticker mapping is incomplete for ADRs / cross-listed / ETFs.** Mitigation: MVP roster in [SPEC §8.3](../../docs/SPEC.md) covers NSE-listed + US majors; edge cases handled in M2 with a lookup table if needed. Flag in `-- TODO(M2):` on the ticker CASE.
- **R3. MIC values in Wealthfolio may differ from the ISO 10383 list we assumed (`XNSE` vs `XNSE_ETF`, etc.).** Mitigation: M2 dumps distinct MIC values from a live DB as its first step before executing; the CASE then becomes data-driven.
- **R4. `daily_account_valuation` decimals stored as TEXT cause silent precision loss on `CAST AS REAL`.** Mitigation: v1 accepts the precision trade; M2 evaluates `printf('%.6f', ...)` vs Python-side `Decimal` parsing. Called out inline.
- **R5. `get-portfolio-twr` daily-aggregate approach assumes no weekend/holiday gaps in `daily_account_valuation`.** Mitigation: wrapper forward-fills. Documented as a wrapper responsibility, not a SQL concern.
- **Open:** none that block drafting. All five user-confirmed decisions are captured above.
