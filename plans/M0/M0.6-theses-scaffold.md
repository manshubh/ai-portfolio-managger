### 2. Ticker normalization rule

For each row after the CSV header:

- Split `Company Name,NSE:SYMBOL` on comma; take `SYMBOL` from the second field after the `NSE:` prefix.
- Emit yaml key `SYMBOL.NS`.
- Preserve hyphens: `NSE:BAJAJ-AUTO` → `BAJAJ-AUTO.NS`. yfinance supports dashed Indian tickers; stripping the dash (`BAJAJAUTO.NS`) resolves to a different or missing symbol.
- Do not uppercase/lowercase — the source symbols are already uppercase.
- Do not strip any other characters. If a future row contains a symbol with a digit (e.g., `M&M`, not currently in the sample), the `.NS` suffix rule still applies verbatim.

The rule assumes all 52 source rows are `NSE:`-prefixed. All 52 are, verified via `grep -c '^[^,]*,NSE:' docs/sample_india.csv` → 52. No `BSE:` prefixes, no foreign exchanges; if the roster later grows a `BSE:SYMBOL`, that row needs a separate rule (yfinance uses `.BO` suffix for BSE) — out of scope for M0.6.

Shape rationale:

- **Top-level key `theses:`** — matches SPEC §7.1.2 example verbatim. M2's merge code (per SPEC §6.1 line 860) will load `yaml.safe_load(open(path))['theses']`.
- **Quoted empty string `""` (not bare/null).** Keeps every value typed as `str`. A `yaml.safe_load` on a bare `TICKER:` produces `None`, and downstream M2 code would then need a `None → ""` coercion that isn't present in the SPEC example. Using `""` removes that ambiguity. Phase 1 snapshot column `thesis` is typed as string per SPEC §7.1.1 — an unquoted `None` coming out of the merge would force a defensive fix in the snapshot writer. Cheaper to be explicit here.
- **Inline `# Company Name` comment** — survives `yaml.safe_load` (comments are stripped). Purpose is for the human editor, not the parser. Preferred over a separate `company_names` block or a prefix-line comment because the association is un-lose-able if the user re-sorts or filters the file. Column-aligned on the `""` token for readability (longest key is `KOTAKBANK.NS` + `: ""` = 18 chars → pad to 18 before `#`; I'll set the pad at 19 for one-space breathing room). The alignment is cosmetic — not verified mechanically; if it drifts later during edits, that's fine.
- **Alphabetical by ticker key.** The source CSV is name-alphabetical (roughly); ticker-alphabetical is the canonical form a user editing the file will want when searching for "does `X.NS` have a thesis yet?" Also stable under ticker renames that change the display name.
- **Header comment block (4 lines).** Points the reader at the SPEC section, the source CSV, and SPEC §19.13 invariant. No frontmatter fences (this is not a Markdown file with YAML front-matter; it's a plain YAML document).

### 4. ETF entries carried through

Three ETFs (`ITBEES.NS`, `NEXT50IETF.NS`, `NIFTYBEES.NS`) appear in the roster with the inline comment suffixed `(ETF)`. The suffix is a note to the user and to M3's implementer — not a schema field. Per SPEC §7.1.2 and the bd issue's acceptance ("values may be empty strings"), carrying ETFs with empty theses is compliant. M3's `scoring-engine` is responsible for recognizing that fundamental metrics like ROE / promoter holding are not applicable to ETFs and skipping the scoring pass for those rows (or producing `instrument_type=ETF` rows that bypass fundamentals-based gates). M0.6 deliberately makes no such decision.

`SBICARD.NS` (SBI Cards & Payment Services) is a single-stock equity, not an ETF despite the similar name — kept without the `(ETF)` marker.

### 5. Implementation mechanics

Two options, both acceptable; I'll default to (a):

- **(a) Hand-written file.** Count is 52 — a single Write operation to `input/india/theses.yaml` with the full content block above. Deterministic, easy to review in the diff, no script dependency. Preferred.
- **(b) Generator script.** A short Python one-liner reads the corrected CSV and emits the YAML. Adds a permanent script for a one-shot job, and the result is the same text. Rejected.

If a future roster refresh is needed (more than ~10 new tickers at once), option (b) becomes attractive — but that's a separate deliverable, not M0.6.

## Risks and open questions

- **R1. These 52 tickers are a sample, not the user's actual holdings.** Per user on 2026-04-23: the CSV is a ticker-only sample, and the actual Wealthfolio-tracked portfolio may differ. Implication: once Wealthfolio is populated (M0.1 + M2), the `export-snapshot` merge will produce rows for holdings whose keys may or may not be in this file, and keys in this file may not correspond to held positions. SPEC §19.13 makes both directions safe (missing thesis → empty; key for non-held ticker → ignored by the snapshot merge). The file is scaffolding, not state, so this is not a defect. Revisit once the real portfolio CSV is imported into Wealthfolio and diffed against this roster — that reconciliation is a separate, optional follow-up.
- **R2. `JSWDULUX` symbol may not survive the JSW Paints rebrand on the exchanges.** The user stated the new company name is "JSW Paints" but kept the ticker direction on `JSWDULUX`. If NSE renames the symbol (standard practice after a corporate action), this key will be wrong. Out of scope to verify live; mitigation is a comment in the bd close-out ("flag if M0.3 smoke-test returns no data for JSWDULUX"). Detection path: M0.3's India MCP smoke test — if a future run adds `JSWDULUX.NS` to the cell matrix and tools return empty, that's the signal.
- **R3. `NEXT50IETF.NS`, `NIFTYBEES.NS`, `ITBEES.NS` are ETFs; M3 may choke on them.** Known and deferred. M0.6 scaffolds; M3 decides how to handle. If M3 can't route ETFs, it adds either a per-ticker `skip: etf` flag in its own config or an `instrument_type` lookup step. No M0.6 rework.
- **R4. The "alphabetical by ticker" ordering puts `BAJAJ-AUTO.NS` before `BANDHANBNK.NS` because `-` (ASCII 0x2D) sorts before letters.** This is Python's default `sorted()` behavior. Acceptable — consistent, stable, and the user's editor will follow the same order when they re-sort. Not a locale-sensitive sort.
- **R5. The inline-comment alignment may drift as the user adds thesis text.** Once a value becomes a multi-word string, the ` # Company Name` comment at column 19 becomes pushed out or re-wrapped. Not a bug — comments are free-form, and `yaml.safe_load` ignores them. Flagging only because someone reading the file later may wonder why alignment got messy; the answer is "because values grew past the padding width." No enforcement needed.
- **Open:** none remaining for execution. All four user clarifications (CSV row fixes, ETF handling, sample-vs-real posture, yfinance format confirmation) resolved on 2026-04-23.
