## Approach

### Top-level structure

`THIRD_PARTY.md` is a single document with five sections in this order:

1. **Header** — one-paragraph purpose, the two-axis taxonomy (vendored vs invoked), and how the file is updated.
2. **Ported source code (vendored under `skills/`)** — code we copy in. SPEC §6.3 governs.
   - §2.1 `virattt/ai-hedge-fund` @ `0f6ac48...`
3. **Invoked external tools (no source vendored)** — code we run as black boxes via MCP / shell. SPEC §19.9 governs.
   - §3.1 `finstack-mcp` 0.10.0
   - §3.2 `nsekit-mcp` 0.0.22
   - §3.3 `nse-bse-mcp` 0.1.5
4. **Runtime prerequisites** — tools the host machine needs for the invocation in §3 to start, but not Python/JS deps of this repo.
   - §4.1 `uv` (provides `uvx`)
5. **External data consumers** — software whose data we read but whose code we never touch.
   - §5.1 Wealthfolio v3.2.1 (consumed-but-not-vendored, no license body inlined per scope decision 6)

The taxonomy matters: §2 entries carry license obligations because we redistribute their code; §3 entries carry version pins so behavior is reproducible; §4 is a runtime sanity check; §5 is documentation that the dependency exists at all.

#### License-text capture mechanism

For finstack-mcp and nse-bse-mcp, M0.4 must capture the MIT text at pin-time (not link to it). At write-time, fetch each upstream LICENSE:

```bash
curl -sS https://raw.githubusercontent.com/finstacklabs/finstack-mcp/v0.10.0/LICENSE
curl -sS https://raw.githubusercontent.com/bshada/nse-bse-mcp/v0.1.5/LICENSE
```

If either upstream uses a tag prefix other than `vX.Y.Z` (e.g. unprefixed), the plan's verification step catches it and the URL is corrected. If a `LICENSE` file is missing on either repo (echoing the ai-hedge-fund situation), the implementation **stops and surfaces the discrepancy to the user** before writing — same posture as M0.2 took. Captured text is inlined verbatim under each section, with attribution noting the URL it was fetched from and the date.

### Update protocol (header §6 of THIRD_PARTY.md)

The file should be self-documenting on who edits what and when. Concrete rules:

- **§2.1 `Ported?` cells** — owned by M3; updated when each port file is committed under `skills/scoring-engine/`. The commit message references this row and the SHA cell becomes the ported-file's local SHA.
- **§3.x version pins** — owned by whichever bd ticket bumps the MCP version (typically a bd ticket of type `dependency-bump`). Pin bump = update §3.x entry + re-fetch license body + add a row to §"Revision log".
- **§4 uv** — bumped only when MCP install instructions stop working with the host's `uv`; not on a schedule.
- **§5 Wealthfolio** — bumped when `config/wealthfolio.md` version pin changes (separate bd ticket per SPEC §19.14).
- **Revision log** — a plain table at the bottom: date | section | change. Each edit appends one row.

## Risks and open questions

- **R1. Upstream MCP `LICENSE` files have moved or use unexpected tag prefixes.** Mitigation: implementation step 2 (license fetch) is explicit; if either `curl` 404s, stop and surface to the user before writing. Same posture as M0.2 used for ai-hedge-fund.
- **R2. Astral changes `uv`'s license posture.** Low risk; even if it shifts to single Apache-2.0, the entry's posture (no inlined body, link upstream) is unchanged. Verification step does not check `uv`'s license body, only that the entry exists.
- **R3. M3 renames port-target local paths.** This plan's §2.1 mapping is *proposed*. M3 owns the final mapping at port time; if it diverges, M3 updates the row. No M0.4 rework needed.
- **R4. Wealthfolio license becomes a redistribution concern.** Out of scope today (we don't redistribute). If the project ever ships as a packaged product, §5.1 is the place to revisit and likely escalate to a proper license body. A bd follow-up is *not* needed at M0.4 close — the §5.1 wording itself surfaces the boundary.
- **R5. `THIRD_PARTY.md` and `config/mcp-servers.md` drift on the NseKit wording.** Mitigation: the verification table `grep -F` check on the verbatim string would fail on drift. Update protocol §6 names `config/mcp-servers.md` as the canonical source; if the wording changes there, M0.4's bd issue is reopened (or a follow-up filed) to re-sync.
- **Open:** none remaining for execution. All four user clarifications (Wealthfolio body, `uv` inclusion, port-target pre-population, NseKit wording) resolved 2026-04-22.
