## Design decisions

| Decision | Chosen | Rationale |
|---|---|---|
| File format | YAML front-matter + prose (same structure as `input/india/philosophy.md`) | M6 reads the file programmatically — "configurable via `config/benchmark-tickers.md`" per implementation.md M6 implies the file is the config, not just docs |
| Market coverage | Both `india` and `us` | US default (`^GSPC`) is uncontroversial; deferring it would leave a `TODO` stub that blocks M6's US path |
| Override mechanism | CLI `--benchmark-ticker` only (SPEC §18.5) | Env vars add surface for a 2-value config; CLI is sufficient and already specified |
| Price-return labeling | Both tickers labeled accurately in config | Technically correct; weekly reports pick up the label from YAML per market. SPEC §14.3 mandates the label in the India report — US report mirrors the treatment even though SPEC is silent on it |
| Alternatives | One-liner per market | Minimal surface to maintain; total-return alternatives are niche at MVP |
| Wealthfolio cross-ref | Short paragraph (§14.4) | Prevents future confusion between two alpha numbers a reader will inevitably see |

## YAML front-matter schema

```yaml
---
benchmarks:
  india:
    ticker: ^NSEI
    name: Nifty 50
    return_type: price-return
    label: "price-return (no dividend reinvestment)"
    source: yfinance
    alternatives:
      total_return: ^NSEI.TRI  # NSE Total Return Index — not on yfinance; prefer NIFTYBEES (NAV-based) if needed
  us:
    ticker: ^GSPC
    name: S&P 500
    return_type: price-return
    label: "price-return (no dividend reinvestment)"
    source: yfinance
    alternatives:
      total_return: ^SP500TR   # available on yfinance; use if dividends matter to alpha comparison
---
```

**M6 read pattern** (using the same `---` split heuristic established in M0.5):

```python
import yaml

def load_benchmark_config(path="config/benchmark-tickers.md"):
    raw = open(path).read()
    _, front, _ = raw.split("---", 2)
    return yaml.safe_load(front)["benchmarks"]

def get_ticker(market: str, override: str | None = None) -> dict:
    if override:
        return {"ticker": override}
    cfg = load_benchmark_config()
    return cfg[market]  # keys: ticker, name, return_type, label, source
```

The CLI `--benchmark-ticker` flag (SPEC §18.5) passes its value as `override`; when absent, M6 reads from this file.

## Prose body outline

The prose body (below the closing `---`) follows the structure of `config/wealthfolio.md` and `config/mcp-servers.md`:

### § 1 — What this file is

One paragraph. This file declares the default benchmark ticker for each market. `skills/benchmark/benchmark.py` (M6) reads the YAML front-matter at runtime. The `--benchmark-ticker` CLI flag (SPEC §18.5) overrides the default for a single run.

### § 2 — Default benchmarks

Prose table mirroring the YAML:

| Market | Ticker | Index name | Return type | yfinance available |
|---|---|---|---|---|
| `india` | `^NSEI` | Nifty 50 | Price-return | Yes |
| `us` | `^GSPC` | S&P 500 | Price-return | Yes |

### § 3 — Override for a single run

Short. Quote the SPEC §18.5 CLI signature:

```
benchmark \
  --market india \
  --benchmark-ticker ^NSEI \
  --windows 1w,1m,3m,1y,3y,inception
```

When `--benchmark-ticker` is given, `config/benchmark-tickers.md` is not read. To change the **permanent default**, edit the `ticker` field in the YAML front-matter of this file.

### § 4 — Price-return limitation

One short paragraph. Both `^NSEI` and `^GSPC` are **price-return** benchmarks on Yahoo Finance: they track index price only, excluding dividend reinvestment. This understates the total return of either index, so the alpha vs. a total-return benchmark would be lower. Weekly reports stamp the `label` field from the YAML front-matter adjacent to each market's alpha rows (SPEC §14.3 mandates this explicitly for `^NSEI`; the US report mirrors the treatment).

One-liner alternatives:
- India total-return: `^NSEI.TRI` (NSE Total Return Index) is not on yfinance at MVP; `NIFTYBEES` (NAV-adjusted) is the practical substitute.
- US total-return: `^SP500TR` is available on yfinance and can be used as a drop-in.

To switch, edit `ticker` in the YAML and confirm yfinance resolves the new symbol before closing.

### § 5 — Wealthfolio "Beat the Market?" (SPEC §14.4)

Two sentences. Wealthfolio has a built-in "Beat the Market?" panel that natively compares account performance to S&P 500 or any configurable ETF. That comparison is transaction-derived and is the **primary visual benchmark** at MVP. The number this `benchmark` skill produces is derived from `daily_account_valuation` snapshots and is intended for programmatic weekly-report inclusion and ledger storage — the two will differ and that is expected.

### § 6 — Version / upgrade policy

One sentence. If a default ticker changes (e.g., migrating to a total-return variant), bump `version` in the YAML, update this file, and file a `bd` ticket before changing production runs — stale `benchmark_alpha_*` ledger rows computed against the old ticker become non-comparable.

## Risks and open questions

- **R1. `^NSEI.TRI` (NSE Total Return Index) is listed as an alternative but may not resolve on yfinance.** Mitigation: prose explicitly warns "not on yfinance at MVP" — the alternative is flagged for awareness, not implied to be usable without verifying. M6's implementation must call `yfinance.Ticker(ticker).history(...)` and surface a clear error if the symbol fails to resolve; that is M6's concern, not this config file's.
- **R2. `^SP500TR` availability on yfinance may change.** Same mitigation — this file documents the intent; M6 validates at runtime.
- **R3. SPEC §14.3 only mandates the price-return label for `^NSEI`.** Extending the label to `^GSPC` in weekly reports is implied by the config design (US report uses `^GSPC`; both are price-return; label comes from YAML). If SPEC is later updated to mandate the `^GSPC` label too, no config change is needed — only M6's markdown renderer changes. Low risk.
- **Open:** None. All design decisions resolved above.
