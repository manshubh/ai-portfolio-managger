## Scope

In scope for this subtask:

1. Re-query upstream `main` HEAD on the day of pinning; compare against the research SHA (`0f6ac48...`, fetched 2026-04-18).
2. Decide the final pin SHA and record rationale.

- Resolving the missing-LICENSE question with upstream. The decision here is to proceed with a documented discrepancy (see [Approach › License handling](#license-handling)).

## Approach

### SHA selection

The M0-investigation notes pinned `0f6ac487986f7eb80749ed42bd26fb8330c450db` on 2026-04-18. Per the planning decision to re-query HEAD on pin day, implementation re-runs:

```bash
curl -sS -H "Accept: application/vnd.github+json" \
  https://api.github.com/repos/virattt/ai-hedge-fund/commits/main \
  | jq -r '.sha, .commit.author.date, .commit.message'
```

**Selection rule:**

1. If HEAD equals the research SHA → pin `0f6ac48...`.
2. If HEAD is ahead → run a compare (`/compare/<research_sha>...main`) and inspect the `files` list. If no file under `src/agents/` in the M3 port list is touched, pin HEAD. Otherwise read the diffs of touched port-list files; pin HEAD only if the changes are additive or non-behavioral. Otherwise pin the last SHA before the regressing change and log rationale.
3. If HEAD is behind (force-push) → treat as a signal and stop; surface to user before proceeding.

**Planning-time signal** (as of 2026-04-21, UTC+0): `GET /compare/0f6ac48...main` reports `ahead_by: 0`, `behind_by: 0` — the repo has not moved in the 3 days since the investigation. Absent a surprise at implementation time, the final pin is **`0f6ac487986f7eb80749ed42bd26fb8330c450db`** (commit date 2026-04-17T21:29:41Z, commit message "Create initial data layer").

### License handling

**The upstream state:** README states *"This project is licensed under the MIT License — see the LICENSE file for details."* However `GET /repos/virattt/ai-hedge-fund/contents/LICENSE` returns 404, and the GitHub repo API's `license` field is `null`. The author has asserted MIT in prose but has not shipped the LICENSE artifact.

**The decision** (chosen at plan time): proceed with the pin, capture standard MIT boilerplate as the pin-time license snapshot, and document the discrepancy prominently. Rationale:

- 56k+ star project with 2+ years of public open-source intent; MIT assertion is in the README, commit history, and project website.
- The risk of the author disputing MIT later is low and mitigated by our attribution (SHA + file list + verbatim MIT text in `THIRD_PARTY.md`).
- Escalating to an upstream issue / fork / author email is higher friction and not justified by the current risk.

**What we capture:** standard MIT template text with the author line `Copyright (c) 2024 virattt` (the GitHub username, in lieu of a named author from a LICENSE file). If the author publishes a LICENSE file later with a different copyright line, M0.4 updates `THIRD_PARTY.md` and this manifest becomes historical.

**What the manifest makes explicit:**

- The repo has **no** `LICENSE` file at the pinned SHA.
- GitHub's API `license` field is `null`.
- The MIT claim comes from `README.md` prose only.
- Our MIT text is the canonical `opensource.org/license/mit/` template with a reconstructed copyright line.
- If we later want zero-ambiguity license posture we should open an upstream issue; a `bd` follow-up task is filed for this if the user wants it (not required for M0.2 close).

### Manifest file — `research/ai-hedge-fund-pin-<shortsha>.md`

Following the M0.1 convention of a version-stamped research artifact. Shortsha is the first 7 chars (`0f6ac48`). Sections, in order:

1. **Pin metadata** — full SHA, short SHA, commit date, commit message first line, pin date (today), upstream URL `https://github.com/virattt/ai-hedge-fund/commit/<sha>`.
2. **Re-query log** — the HEAD SHA observed at re-query time, the compare result (`ahead_by`, `behind_by`), and the selection rule that produced the final pin. If research SHA was used unchanged, record that too.
3. **Rationale** — 3–5 bullets: why this SHA over HEAD / over an older tag, green-main heuristic used, and any behavioral concerns.
4. **Port targets at this SHA** — the 10-file table with HTTP status from the verification check, each cell linking to the raw blob at the pinned SHA.
5. **License** — three parts:
   - Status line: `README: claims MIT · LICENSE file: missing · GitHub API license: null`.
   - Verbatim MIT text with reconstructed copyright line.
   - One-paragraph disclosure explaining the discrepancy, to be copied into `THIRD_PARTY.md`.
6. **Consumer map** — short list of downstream milestones that read this manifest: M0.4 (THIRD_PARTY.md) and M3 (scoring-engine port).

The file is ≤ ~150 lines. No binary content. No copies of upstream source — only URLs.

## Risks and open questions

- **R1. Upstream force-push between plan-time and implementation.** The selection rule catches `behind_by > 0` and stops. No silent downgrade.
- **R2. Author later disputes MIT or publishes a non-MIT LICENSE file.** Mitigation: the pin is to an immutable SHA; removing the license claim from `main` does not retroactively relicense the code at `0f6ac48`. Our `THIRD_PARTY.md` entry cites the commit and carries the MIT text; we are on defensible ground even if upstream changes posture.
- **R3. Port-target file renamed or deleted in a future upstream commit before M3 starts.** Not a risk for *this* pin — the SHA is immutable and the verification table confirms all 10 files exist at it. If M3 decides to refresh the pin, that is a new `bd` task, not M0.2.
- **R4. "Green main" is unenforceable.** The upstream repo has no CI badge and no release tags; we cannot mechanically verify main is green. Mitigation: the 10-file HTTP-200 check is our proxy — the files we care about exist and are syntactically retrievable. Deeper verification (importability, test-passing) is an M3 concern, not M0.2.
