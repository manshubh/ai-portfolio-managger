# M0.9 — Docker environment for skill layer

> bd: [ai-portfolio-manager-zgn](../../.beads/) · parent epic: M0
> Related: [docs/implementation.md](../../docs/implementation.md) · [docs/SPEC.md](../../docs/SPEC.md)

## Scope

In scope for this subtask:

1. Identify an appropriate base Docker image covering Python, `sqlite3`, `shellcheck`, and `yaml` parsing tools.
2. Author `Dockerfile` at the repo root to containerize the Python/Shell skill layer.
3. Author `docker-compose.yml` at the repo root.
4. Configure volume mounts in `docker-compose.yml` to mount the local repository and the Wealthfolio SQLite DB as read-only.
5. Provide a host-level wrapper script (`bin/run-skill`) to allow the Claude agent to easily execute commands inside the container without writing out `docker compose run` every time.

Out of scope:

- Containerizing Claude Desktop or the Wealthfolio UI. Both stay natively on the host per [SPEC §17](../../docs/SPEC.md).
- Running the `wealthfolio-query` tool directly against a real database yet (execution and testing happens in later milestones).
- Writing any Python/Shell skills themselves — this milestone only sets up the execution environment.

## Design decisions

| Decision | Chosen |
|---|---|
| Base Image | `python:3.11-slim` (or similar stable Debian slim base) — Provides a modern Python environment while keeping the image size small. We will bypass PEP 668 with `--break-system-packages` for simplicity. |
| System Packages | `build-essential` (for future C-extensions), `sqlite3`, `shellcheck`, `jq` (often useful for shell/JSON tasks). `python3-yaml` or `pip install pyyaml` for YAML parsing. |
| Wealthfolio DB Mount | Read-only bind mount to the host's Wealthfolio `ledger.db`. The host path will be passed as an environment variable (`WEALTHFOLIO_DB_PATH`). The container does not require this DB to exist to boot (e.g., Phase 2 research runs fine without it). |
| Repository Mount | The repo root is bind-mounted to `/workspace` inside the container. File permissions will be left as default; the user will handle any root-owned cleanup if necessary. |
| Execution Wrapper | `bin/run-skill` will be a thin bash script on the host passing commands to `docker compose run --rm skills-env "$@"`. |
| Network | Default bridge network is sufficient; no inbound ports need to be exposed. |

## Approach

### Dockerfile

```dockerfile
# Dockerfile
FROM python:3.11-slim

# Install system dependencies
RUN apt-get update && apt-get install -y \
    build-essential \
    sqlite3 \
    shellcheck \
    jq \
    && rm -rf /var/lib/apt/lists/*

# Install Python dependencies globally for the skill layer
# (Will be populated further in later milestones like M3, M4, M6)
RUN pip install --no-cache-dir pyyaml --break-system-packages

WORKDIR /workspace

# Set default command
CMD ["bash"]
```

### docker-compose.yml

```yaml
# docker-compose.yml
version: '3.8'

services:
  skills-env:
    build: .
    volumes:
      # Mount the repository into the container
      - .:/workspace
      # Mount the Wealthfolio DB read-only. The path is provided via an environment variable.
      - ${WEALTHFOLIO_DB_PATH:-/Users/mrihal/Library/Application Support/Wealthfolio/ledger.db}:/wealthfolio/ledger.db:ro
    environment:
      - WEALTHFOLIO_DB=/wealthfolio/ledger.db
    # Keep the container running for interactive use or exec
    command: tail -f /dev/null
```
*Note: The default macOS path for Wealthfolio is shown as a fallback. Users can override `WEALTHFOLIO_DB_PATH` in a `.env` file.*

### Host Execution Wrapper (`bin/run-skill`)

```bash
#!/usr/bin/env bash
# Runs a command inside the skills-env Docker container.
# Usage: bin/run-skill skills/scripts/claim-stocks.sh

if [ "$#" -eq 0 ]; then
    echo "Usage: $0 <command> [args...]"
    exit 1
fi

exec docker compose run --rm skills-env "$@"
```

## Verification

| Criterion | Verification |
|---|---|
| `Dockerfile` exists | `test -s Dockerfile` |
| `docker-compose.yml` exists | `test -s docker-compose.yml` |
| Image builds successfully | `docker compose build` succeeds with no errors |
| Tools are present | `docker compose run --rm skills-env bash -c "python --version && sqlite3 --version && shellcheck --version && python -c 'import yaml'"` returns 0 |
| Wealthfolio DB mount is read-only | `docker compose run --rm skills-env bash -c "touch /wealthfolio/ledger.db"` fails with a Read-only file system error |

## Risks and open questions

- **R1. Wealthfolio DB path varies heavily by OS.** Mitigation: Use an environment variable `WEALTHFOLIO_DB_PATH` in `docker-compose.yml` and provide a `.env.example` file that explains how to configure it per platform.
- **R2. Docker mounts an empty directory if the host file doesn't exist.** Mitigation: If the user hasn't set up Wealthfolio yet, Docker will create an empty `ledger.db/` directory on the host to satisfy the mount. Skills querying the DB will fail with a SQLite error, but the container will still boot and allow non-DB tasks (like Research) to run.
- **R3. File permissions on mounted volumes.** Mitigation: We accept that files created by the container will be owned by root. The user handles cleanup natively on the host.
