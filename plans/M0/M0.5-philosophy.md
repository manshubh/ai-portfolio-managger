## Approach

### 1. YAML front-matter composition

The front-matter is a single YAML document between two `---` fences at the top of the file. Top-level keys, in the order they should appear:

```
market
version
draft_notice           (non-schema: a short string flagging draft status)
goals                  (non-schema extension — §1.3)
thresholds             (SPEC §7.6 — §1.1)
preferred              (non-schema extension — §1.2, parallel shape to thresholds)
position_sizing        (SPEC §7.6)
risk                   (SPEC §7.6 — extended with small_cap variant and debate triggers)
sector_exceptions      (SPEC §7.6)
personas_enabled       (SPEC §7.6)
persona_rotation       (SPEC §7.6)
```

The ordering matches the SPEC §7.6 example top-to-bottom so a reader can diff the two side-by-side. Non-schema extensions (`draft_notice`, `goals`, `preferred`, two new risk keys) live **below** the fields they most closely relate to, so dropping them later is a single localized delete.

#### 1.2 `preferred` block — shape parallel to `thresholds`

The `preferred` block mirrors `thresholds` key-for-key, sourced from the "Preferred" column of the same two prose tables. Suffix is `_preferred` (not `_min` / `_max`) so it is obviously distinct:

```yaml
preferred:
  non_financial:
    roe_preferred: 18
    roce_preferred: 18
    de_preferred_max: 0.5
    promoter_preferred: 55
    pledge_preferred_max: 0
    revenue_cagr_3y_preferred: 15
    profit_cagr_3y_preferred: 15
    fcf_positive_years_preferred: 3      # prose: "Consistently positive for 3+ years"
    mcap_preferred_cr: 20000
  banking_nbfc:
    roa_preferred: 1.5
    roe_preferred: 15
    nnpa_preferred_max: 1.0

```

`preferred` is **not part of SPEC §7.6** — it is an M0.5 extension. The plan is not proposing a SPEC change; M3 can ignore this block if the scoring-engine port does not use it. It is retained so no prose content is silently lost during migration.

#### 1.3 `goals` block — horizon + alpha target

One new block capturing two prose statements that have no SPEC §7.6 home:

```yaml
goals:
  horizon_years_min: 10                   # prose: "Long-term wealth creation (10+ year horizon)"
  alpha_target_pct_min: 3                 # prose: "Beat Nifty 50 returns by 3-5% annually"
  alpha_target_pct_max: 5

```

Benchmark ticker itself lives in M0.8's `config/benchmark-tickers.md`; this block only captures the *magnitude* of the target, keeping concerns separated.

#### 1.5 `draft_notice` — flag the file's status

A single scalar string at the top of the YAML:

```yaml
draft_notice: "Draft — numerical thresholds are starting points; the authoritative narrative is the prose body below. User will iterate as portfolio data accumulates and as scoring-engine calibration exposes mismatched thresholds. Schema is per SPEC §7.6 with non-breaking extensions (preferred, goals, two risk keys) flagged in plans/M0/M0.5-philosophy.md §1."
```

This exists so a future reader (or a future Claude session) reading only the file sees the "this is a draft" signal without having to cross-reference the plan. It is a free-form key; M3 ignores it.

### 4. YAML parse and spot-check

Acceptance criterion from the bd issue, verbatim:

```bash
python3 -c "import yaml; doc = open('input/india/philosophy.md').read().split('---')[1]; parsed = yaml.safe_load(doc); assert 'thresholds' in parsed; assert parsed['thresholds']['non_financial']['roe_min'] == 12; print('OK')"
```

The issue's exact one-liner uses `chr(45)*3` in place of the literal `---` sentinel to avoid shell quoting; either form is acceptable. A successful run prints `OK` and exits 0.

### 5. Why the schema extensions are safe

`thresholds`, `position_sizing`, `risk`, `sector_exceptions`, `personas_enabled`, `persona_rotation` — all six are **SPEC §7.6 schema keys** and must keep their names and numeric types. The scoring-engine port (M3) reads these.

`goals`, `preferred`, `draft_notice`, and the two new `risk.*` keys are **extensions**. They are additive (do not rename or retype any SPEC field) and scoped (all net-new top-level is `goals` + `draft_notice`). If M3 uses a strict YAML parser that rejects unknown keys, M3 can either add these to the schema or ignore them at load time — both options are cheap, and neither forces a SPEC change today. The tradeoff is explicit: **we preserve prose content** (Q1/Q2 answer: option **b**) at the cost of two extensions + one mirrored block. User accepted this.

## Risks and open questions

- **R1. SPEC §7.6 evolves (v3 schema) between M0.5 and M3.** Mitigation: the file sets `version: 2` in the front-matter. If M3's scoring-engine parser bumps the schema, M3's own plan owns the migration — M0.5 is not responsible for forward-porting.
- **R2. M3 rejects the `preferred`, `goals`, `draft_notice` extensions.** Low risk. M3 can choose `yaml.safe_load` (ignores unknown keys) or a stricter parser; if strict, M3 extends its schema or pops the keys at load. No M0.5 rework either way.
- **R3. User edits `docs/philosophy.md` after M0.5 closes, not realizing it was deleted.** Mitigated by the deletion happening as part of this subtask and the session-close hand-off flagging the new canonical path. Deletion is reversible via git until pushed.
- **R4. Prose numeric values drift from YAML.** Known accepted risk. Numbers live in two places now (YAML + prose tables). This is an architectural property of SPEC §7.6, not a defect of M0.5 — the scoring-engine reads YAML as authoritative, the LLM reads prose as authoritative for narrative. If they drift, the scoring-engine's deterministic tables would diverge from the agent's qualitative rationale, and the debate pattern per SPEC §11 would catch the mismatch in the report output. Acceptable at M0 — revisit if it becomes a recurring support issue.
- **R5. The `---` split heuristic in the verification Python breaks if prose contains a `---` line.** Real: the existing `docs/philosophy.md` uses `---` as horizontal rules between sections (prose lines 17, 33, 77, 94, 107, 116, 127, 161, 175). The verification script uses `.split('---')[1]` which correctly extracts the second element (the YAML between the first two `---`s). But any code reading the prose body must use `.split('---', 2)[2]` or a proper front-matter parser (e.g., `python-frontmatter`), not a naive split. M3's parser needs to know this — flag it in M3's plan. No M0.5 change needed; the SPEC §7.6 example does not specify a parser.
- **Open:** none remaining for execution. All six user clarifications (YAML extension posture, `preferred` placement, prose fidelity, old-file deletion, US out-of-scope, plan location) resolved on 2026-04-23.
