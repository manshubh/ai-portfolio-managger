| MCP | Ecosystem | Install command | Investigation version | License |
|---|---|---|---|---|
| `finstack-mcp` | PyPI | `pip install finstack-mcp==<ver>` (or `pipx install finstack-mcp==<ver>`) | 0.10.0 (2026-04-09) | MIT |
| `NseKit-MCP` | PyPI | `uvx nsekit-mcp@<ver>` | 0.0.22 (2026-03-25) | **None** |
| `nse-bse-mcp` | npm | `npx -y nse-bse-mcp@<ver>` (server binds `localhost:3000`) | 0.1.5 | MIT |

Verification step at implementation time:

```bash
pip index versions finstack-mcp
pip index versions nsekit-mcp
npm view nse-bse-mcp version
```

Any delta vs the investigation is logged in the subtask notes and the newer version becomes the pin unless a regression shows in release notes.

### Registration templates

**`.mcp.json` (Claude Code)** and **`.cursor/mcp.json` (Cursor)** share the same JSON shape (`{"mcpServers": {...}}`) so the two files are near-identical. Committing both makes the project work for either IDE without manual fiddling.

Shape (final values filled at implementation time after version pin):

```json
{
  "mcpServers": {
    "finstack": {
      "command": "python",
      "args": ["-m", "finstack.server"]
    },
    "nsekit": {
      "command": "uvx",
      "args": ["nsekit-mcp@0.0.22"]
    },
    "nse-bse": {
      "command": "npx",
      "args": ["-y", "mcp-remote@latest", "http://localhost:3000/mcp", "--allow-http"]
    }
  }
}
```

Deviation note: the bd issue wording says `.claude/settings.json`. Current Claude Code MCP registration lives in `.mcp.json` at repo root; `.claude/settings.json` is for hooks/permissions (already present in this repo with `PreCompact` / `SessionStart` bd hooks). We ship `.mcp.json` instead and leave the existing `.claude/settings.json` untouched. This deviation is called out in the close-note on the bd issue.

### `uvx` runtime prereq (both `finstack` and `nsekit`)

Both `finstack` and `nsekit` are registered via `uvx` (see `.cursor/mcp.json` / `.mcp.json`). If `uv` is not installed, Cursor spawns `uvx ...`, the command is not found, and **both** servers fail startup with the generic "The MCP server errored" message in the MCP panel — no tool list is exposed. `nse-bse-mcp` is unaffected because it uses `npx` (which ships with Node).

Observed on 2026-04-22: a fresh macOS clone of this repo hit this exact failure mode; both servers errored simultaneously until `uv` was installed. After `brew install uv` + a Cursor relaunch, both servers loaded their full tool surface (finstack ≈ 93 tools, nsekit ≈ 116 tools + prompts).

Preflight (macOS):

```bash
brew install uv    # ships `uv` and `uvx`; verify with `uvx --version`
```

Linux / other: `curl -LsSf https://astral.sh/uv/install.sh | sh` (ensure `~/.local/bin` or the reported install dir is on Cursor's spawn PATH — on macOS, Cursor launched from the Dock inherits the login shell PATH; relaunch after install).

**`config/mcp-servers.md` must surface this as the first preflight item**, above the `nse-bse-mcp` server-start step, because it blocks two of the three MCPs. M0.4 also carries this into `THIRD_PARTY.md` (uv is a runtime dependency for MCP registration, not a Python-package dependency of this repo).

### `nse-bse-mcp` HTTP-server prereq

`nse-bse-mcp` runs as a Node HTTP server on `localhost:3000`; the MCP client connects through `mcp-remote`. Before any Cursor/Claude session can use it, the user starts the server in a dedicated terminal:

```bash
npx -y nse-bse-mcp@<ver>   # blocks; leaves localhost:3000 listening
```

`config/mcp-servers.md` documents this as a preflight step. If the smoke test runs without that terminal, all `nse-bse-mcp` tool calls will fail with a connection error — that is expected and not a bug; the doc calls it out.

### Smoke-test matrix

Scope: every tool named in SPEC §18.2 plus the `corporate_events` tool named in §6.2. Three tickers per market; `N/A (market)` cells are skipped intentionally, not failures. The two matrices are run in separate sessions.

**India session** — server prereq: `nse-bse-mcp` running on `localhost:3000`.

| Tool | MCP | RELIANCE.NS | HDFCBANK.NS | INFY.NS | Downstream consumer |
|---|---|---|---|---|---|
| `get_fundamentals` | finstack | probe | probe | probe | M2 aggregator, Phase 2 [Fund] |
| `get_income_statement` | finstack | probe | probe | probe | Phase 2 [Fund], 3y CAGR |
| `get_balance_sheet` | finstack | probe | probe | probe | Phase 2 [Fund], D/E (bank: CAR/CASA) |
| `get_ratios` | finstack | probe | probe | probe | Phase 2 [Valu], P/E, ROE, OPM |
| `get_fii_dii` | finstack | probe | probe | probe | Phase 2 [Fund] institutional flows |
| `nse_equity_quote` | nse-bse | probe | probe | probe | Phase 2 India live quote fallback |
| `equity_history` (~1y) | nsekit | probe | probe | probe | Phase 2 MAs (100d/200d) |
| `corporate_events` (±90d) | nsekit | probe | probe | probe | Phase 1, M5 corp-actions |

*India cell count: 24.*

**US session** — `nse-bse-mcp` localhost server NOT required (tools are NSE-only); registering NseKit + nse-bse in this session is allowed but their tools go uncalled.

| Tool | MCP | AAPL | JPM | MSFT | Downstream consumer |
|---|---|---|---|---|---|
| `get_fundamentals` | finstack | probe | probe | probe | M2 aggregator, Phase 2 [Fund] |
| `get_income_statement` | finstack | probe | probe | probe | Phase 2 [Fund], 3y CAGR |
| `get_balance_sheet` | finstack | probe | probe | probe | Phase 2 [Fund]; bank CAR/tier-1 for JPM |
| `get_ratios` | finstack | probe | probe | probe | Phase 2 [Valu], P/E, ROE |
| `get_sec_filing` (10-K) | finstack | probe | probe | probe | Phase 2 US Tier 1 |
| `get_sec_filing` (10-Q) | finstack | probe | probe | probe | Phase 2 US Tier 1 |

*US cell count: 18. Total across both sessions: 42.*

"probe" = one invocation with the stated ticker; the RAW response is captured verbatim (truncated to the first 40 lines if huge) into the scratch paste file.

**Pass criterion per cell:** non-empty structured response with at least one expected field (e.g., `get_fundamentals` returns `revenue` or `market_cap`; for a bank, `get_balance_sheet` returns `total_deposits` or `tier_1_ratio`). Empty object, error payload, or "N/A / not supported" is a gap → logged in `mcp-gaps.md` with the market column set.

**Bank-specific expectations.** `HDFCBANK.NS` and `JPM` are the only cells where `get_balance_sheet` / `get_ratios` need banking-metric fields (CASA%, GNPA%, NIM% for India; tier-1, NIM for US). If finstack returns only generic non-financial shape for the bank, that is a gap — M3 needs distinct fields for the `banking_nbfc` scheme. Logged with severity `high` because it directly blocks M3's HDFCBank fixture.

### Smoke-test execution protocol

The user executes, the agent formalizes. **Two sessions, one per market**, so that persona / context bleed between markets cannot corrupt the test (and because reports are market-scoped downstream — this mirrors the runtime topology).

1. **Agent (before user sessions):** writes two paste-ready prompt scripts — `plans/M0/M0.3-smoke-script-india.md` and `plans/M0/M0.3-smoke-script-us.md` — each containing its market's matrix rendered as literal prompts the user pastes into a fresh Cursor/Claude window. Example line: `> Use finstack get_fundamentals on RELIANCE.NS and paste the raw JSON (first 40 lines if long).`
2. **User, India session:**
   a. Starts `nse-bse-mcp` server on `localhost:3000`.
   b. Opens a fresh Cursor/Claude window, confirms the three MCPs are loaded.
   c. Pastes India prompts top-to-bottom; captures every response verbatim into `temp/mcp-smoke-raw-india.md` (scratch, gitignored).
3. **User, US session:**
   a. Fresh window (can keep `nse-bse-mcp` server stopped; finstack is the only needed MCP for US).
   b. Pastes US prompts; captures into `temp/mcp-smoke-raw-us.md`.
4. **Agent (after user completes both sessions):** parses each scratch file, writes the final `research/mcp-smoke-test-india.md` and `research/mcp-smoke-test-us.md` (summary + per-cell pass/fail + verbatim excerpts), and appends any failing / thin cells to the single `research/mcp-gaps.md` with a `market` column. Closes the bead.

### `config/mcp-servers.md` outline

Sections, in order:

1. **What these MCPs are** — one paragraph referencing SPEC §15.2 tier system.
2. **Per-MCP table** — name, version pin, install command, license status, default tools used by this project, known limits.
3. **Registration** — point at `.mcp.json` (Claude Code) and `.cursor/mcp.json` (Cursor); explain the shared `mcpServers` shape; note that `~/.claude.json` / `~/.cursor/mcp.json` are user-scoped alternatives.
4. **Preflight** — (a) install `uv` (`brew install uv` on macOS) so `uvx` is on Cursor's spawn PATH; without it, `finstack` and `nsekit` both error on startup. (b) start the `nse-bse-mcp` `localhost:3000` server before the agent session.
5. **NseKit license posture** — two-sentence risk statement (sourced verbatim into `THIRD_PARTY.md` by M0.4).
6. **Smoke-test references** — two separate sessions (India + US); link to `research/mcp-smoke-test-india.md` and `research/mcp-smoke-test-us.md`; summary mentions the 42-cell total.
7. **Coverage gaps** — single cross-market file `research/mcp-gaps.md` with a `market` column.

Target length: ≤ ~140 lines.

### `research/mcp-smoke-test-{india,us}.md` outline (same shape, two files)

1. **Run metadata** — date, session (Cursor vs Claude Code), MCP versions, tickers used (`RELIANCE.NS`/`HDFCBANK.NS`/`INFY.NS` or `AAPL`/`JPM`/`MSFT`).
2. **Per-cell table** — 24 rows (India) or 18 rows (US) from the matrix, each with: pass/fail, response-size bytes, first field observed, verbatim excerpt (first 20 lines of the raw JSON/output).
3. **Summary** — `X / 24 cells pass` (India) or `X / 18 cells pass` (US), mapped to the SPEC §18.2 aggregator requirements. Banking cells called out separately (M3 sensitivity).
4. **Gaps** — one-line pointers into `research/mcp-gaps.md` filtered to this market.

### `research/mcp-gaps.md` outline

Single cross-market file. Per gap row: (market, tool, MCP, ticker, observation, severity, downstream phase affected, bd follow-up ID or "none yet"). Rows are informational only; M5/M14 revisit per SPEC §19.14 + epic M0 risk. Empty file at close-time (`# No gaps observed.`) is an acceptable outcome.

**Banking rows:** gaps on `HDFCBANK.NS` or `JPM` rows where `get_balance_sheet` / `get_ratios` return non-banking shape are marked `severity: high` and get a bd follow-up before close (feeds M3 directly).

## Risks and open questions

- **R1. NseKit 0.0.22 is early-stage and could stop installing between plan-time and smoke-test-time.** Mitigation: the smoke test is the canary — a failing install is logged as a gap and M5 falls back to Yahoo per SPEC §6.2 / bd `ap0`. No downstream escalation required for M0.3 itself.
- **R2. NseKit's missing license becomes a redistribution blocker.** Local-only invocation is the mitigation baked into §5 of `config/mcp-servers.md`; no NseKit code ships in this repo. If the user later wants to distribute the repo as a product, this risk escalates and a fork-or-replace bd ticket is opened.
- **R3. `nse-bse-mcp` requires a foreground server process on `localhost:3000`; easy to forget.** Mitigation: preflight section in `config/mcp-servers.md` + an explicit error-mode note in the smoke-test doc for the case where the server was not running.
- **R4. finstack `get_sec_filing` might not expose the exact signature SPEC §18.2 assumes (`{ticker} {filing_type}`).** Mitigation: the smoke probe captures the real signature; any deviation is flagged in `mcp-gaps.md` and M9 adapts, not SPEC.
- **R5. Package versions drift between plan date (today) and implementation.** Mitigation: the re-query step in Approach §"Package confirmation" is mandatory; the notes field records any version change from the investigation's 0.10.0 / 0.0.22 / 0.1.5 baseline.
- **R6. Smoke session on a different machine than the one running Wealthfolio.** Not a risk for M0.3 — MCPs are independent of Wealthfolio DB access.
- **R7. Banking cells (`HDFCBANK.NS`, `JPM`) reveal that finstack doesn't surface bank-shaped balance sheets.** If both bank cells fail on `get_balance_sheet` / `get_ratios`, M3's `banking_nbfc` scheme cannot be fixture-sourced from the MCPs. Mitigation: the gap is flagged `severity: high` with a bd follow-up; M3 either adds a scraper or uses a hand-curated `tests/scoring-engine/hdfcbank.json` seeded from Screener.in (SPEC §15.3 allows this). This decision is owned by M3, not M0.3, but M0.3 must surface the signal.
- **R8. Three-ticker-per-market smoke takes ~10 minutes per session.** Not a risk, just calibration; the scripts are paste-only so throughput is bounded by MCP latency.
