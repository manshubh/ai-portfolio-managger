## Scope

In scope for this subtask:

1. Confirm Wealthfolio is installed and runnable on this machine.
2. Initialize the SQLite DB file by launching the app once (Diesel migrations run on first start; until the app has been opened, `app.db` does not exist).
3. Dump the schema to `research/wealthfolio-schema-v3.2.1.txt` — DDL only, no row data.
4. Author `config/wealthfolio.md` covering install, DB path, Docker Option A bind-mount, and **holdings-only maintenance guidance**.
5. Record the version pin (v3.2.1, build 20260301.1) for consumption by M0.4 (`THIRD_PARTY.md`).

Out of scope:

- Writing any SQL queries against the dumped schema — that is M0.7.
- Building the Dockerfile / docker-compose.yml — that is M0.9. This plan only specifies the *mount contract* that M0.9 must honor.
- Seeding fixture data — that is M2.

## Approach

### Install state

Wealthfolio is already present at `/Applications/Wealthfolio.app` (confirmed v3.2.1 via `defaults read .../Info.plist CFBundleShortVersionString`). No `brew` reinstall needed. The binary exposes no CLI (`wealthfolio --version` fails) — version is read from the bundle.

### DB initialization

The user reports "no setup done" — implying the app has not been opened. Until first launch, `~/Library/Application Support/com.teymz.wealthfolio/app.db` does not exist because Diesel migrations run at startup.

**Steps:**

1. Launch Wealthfolio.app from Finder or Spotlight.
2. Accept any first-run prompts. If the app requires at least one account to proceed past the welcome screen, create a single dummy account (name: `init`, any broker / currency). This is the minimum state required for migrations to complete and the DB file to materialize. No transactions or holdings needed for M0.1; a populated dummy account is fine.
3. Quit the app before running the schema dump (WAL checkpoint on clean close).
4. Verify `~/Library/Application Support/com.teymz.wealthfolio/app.db` exists and is non-empty.

### Schema dump

```bash
export WEALTHFOLIO_DB="$HOME/Library/Application Support/com.teymz.wealthfolio/app.db"
sqlite3 -readonly "$WEALTHFOLIO_DB" '.schema' > research/wealthfolio-schema-v3.2.1.txt
```

`-readonly` is non-negotiable per SPEC §19 invariant 1 (agent never writes Wealthfolio DB). Use this pattern in the doc as well.

Sanity check the output: at least 28 `CREATE TABLE` statements per research notes; grep for `accounts`, `activities`, `assets`, `quotes`, `daily_account_valuation`, `holdings_snapshots`.

### Docker mount contract (Option A)

The chosen topology:

- Wealthfolio runs natively on macOS (Tauri desktop app; cannot be containerized).
- All Python/shell skill code runs in the Docker container built in M0.9.
- Container mounts the host Wealthfolio app-support directory read-only:

```yaml
# relevant fragment for M0.9's docker-compose.yml
services:
  skills:
    volumes:
      - type: bind
        source: ${HOME}/Library/Application Support/com.teymz.wealthfolio
        target: /wealthfolio
        read_only: true
    environment:
      - WEALTHFOLIO_DB=/wealthfolio/app.db
```

**Caveats to document in config/wealthfolio.md:**

- SQLite WAL files (`app.db-wal`, `app.db-shm`) live next to `app.db`. The mount must cover the whole directory, not just the `.db` file, so a read-only open can see the WAL.
- The user should **quit Wealthfolio before running the agent**. `-readonly` opens tolerate a dirty WAL but cannot checkpoint; a concurrent write from the GUI can produce a momentarily-inconsistent read.
- macOS Docker bind mounts use VirtioFS by default. Acceptable for read-heavy SQLite access; no tuning needed for M0.1.

Alternatives (B snapshot-copy, C host-native, D export-import) were considered and rejected in planning because (a) the DB is rarely written during scan runs and (b) Option A is the simplest contract downstream milestones have to honor.

Sections, in order:

1. **What Wealthfolio is** — one paragraph, plus link to the project repo.
2. **Maintenance mode: holdings-only** — the user maintains current positions and cost basis; transactions, dividends, and corp-action entries are optional. Downstream effects on M5 and long-window alpha are called out here with a pointer to SPEC §5. *(This section depends on bd [ai-portfolio-manager-ap0](../../.beads/) landing; draft it now and reconcile once SPEC is updated.)*
3. **Install** — `brew install --cask wealthfolio`; verify with `defaults read /Applications/Wealthfolio.app/Contents/Info.plist CFBundleShortVersionString`.
4. **First launch and DB initialization** — create a dummy account, quit, verify `app.db` exists.
5. **DB path and override** — default path + `WF_DB_PATH` env var.
6. **Docker mount (Option A)** — the compose fragment above, plus the three caveats.
7. **Quitting-before-scans discipline** — one short instruction block.
8. **Version pin** — v3.2.1 (build 20260301.1); upgrade is a separate bd ticket per SPEC §19.14.

## Verification

Acceptance criteria, each mapped to a check:

| Criterion | Verification |
|---|---|
| Schema dump exists and is committed | `test -s research/wealthfolio-schema-v3.2.1.txt` |
| Dump is schema-only, no row data | `grep -cE '^(INSERT\|COMMIT\|BEGIN)' research/wealthfolio-schema-v3.2.1.txt` returns 0 |
| DB is readable in read-only mode | `sqlite3 -readonly "$WEALTHFOLIO_DB" '.schema'` exits 0 |
| Output matches the committed dump | `diff <(sqlite3 -readonly "$WEALTHFOLIO_DB" '.schema') research/wealthfolio-schema-v3.2.1.txt` is empty |
| Expected core tables present | `grep -cE 'CREATE TABLE.*(accounts\|activities\|assets\|quotes\|daily_account_valuation\|holdings_snapshots)' research/wealthfolio-schema-v3.2.1.txt` returns ≥ 6 |
| `config/wealthfolio.md` authored | `test -s config/wealthfolio.md` and review that all 8 outlined sections are present |
| Version pin recorded | `grep -F '3.2.1' config/wealthfolio.md` returns at least one match |

## Risks and open questions

- **R1. DB doesn't initialize without a dummy account.** Mitigation: plan prescribes creating one and quitting. If Wealthfolio is non-deterministic about when migrations run, surface in the subtask notes and consider removing the dummy account after the dump.
- **R2. Schema drift on future upgrade.** Mitigation: the dump is version-stamped (v3.2.1). Upgrading Wealthfolio is a separate bd ticket per SPEC §19.14.
- **R3. WAL/SHM sidecars absent if app was quit uncleanly.** Mitigation: checklist in `config/wealthfolio.md` tells the user to quit cleanly (File → Quit, not force-quit) before scan runs. Read-only opens still work on a dirty WAL but may miss the last-committed transaction.
- **Open:** none remaining for M0.1 execution. The holdings-only wording coordination with bd ap0 is the only coupling point.
