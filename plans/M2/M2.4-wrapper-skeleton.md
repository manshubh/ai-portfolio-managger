# M2.4 - Wrapper skeleton

## Scope

This plan covers `ai-portfolio-manager-l74.4`: create the minimal `skills/wealthfolio_query/` package that later M2 subcommand tasks plug into.

In scope:
- `skills/wealthfolio_query/__init__.py` package marker.
- `skills/wealthfolio_query/query.py` with named-query parsing, schema-version validation, read-only SQLite open helpers, and a small CLI surface for skeleton dispatch.
- `skills/wealthfolio_query/query.sh` as the user-facing shell entrypoint with `set -euo pipefail`, help text, environment validation, and known-subcommand dispatch.
- Exit-code conventions and error-message prefixing.
- Read-only plumbing for both Python and shell execution paths.

Out of scope:
- Real `export-snapshot` thesis merge behavior (M2.7).
- TWR chained-product calculation and forward-fill (M2.8).
- The four scalar/list subcommand implementations (M2.6).
- `--strict-scope` validation behavior (M2.9).
- README prose (M2.10).
- Fixture data and reusable test scripts (M2.5), except for naming the interfaces those tests will exercise.

## SPEC References

- SPEC §5.3: Wealthfolio access is holdings-only and goes through versioned named SQL.
- SPEC §6.8: `wealthfolio-query` is a read-only wrapper around named parameterized queries.
- SPEC §17: skill layout uses `skills/wealthfolio_query/` and `skills/sql/wealthfolio-queries.sql`.
- SPEC §18.1: `wealthfolio-query --help` exposes the subcommand signatures.
- SPEC §19.2 invariant 11: never write the Wealthfolio DB.
- SPEC §19.2 invariant 14: SQL is pinned to the installed Wealthfolio release and upgrades require explicit review.

## Inputs

M2.4 consumes:
- `skills/sql/wealthfolio-queries.sql`, already cleaned by M2.3 and containing `-- name: <slug>` markers.
- `config/wealthfolio.md`, whose version pin is `v3.2.1` with build `20260301.1`.
- `research/milestones/M2-investigation.md` §1, §2, §8, and §11.
- Bead acceptance criteria from `ai-portfolio-manager-l74.4`.

The shell entrypoint reads the DB path from `WEALTHFOLIO_DB`. If unset, it may resolve the documented macOS default path for native use, but the implementation must still fail with exit 2 when no usable DB file can be opened.

## Approach

Use the investigation's hybrid shape: `query.sh` remains the stable executable surface, while `query.py` owns the reusable Python primitives. This keeps compatibility with SPEC §17's shell wrapper while avoiding duplicated parser and read-only-open logic in later helper modules.

Rejected alternatives:
- Pure shell: workable for scalar commands, but brittle for named-query parsing and later CSV/TWR helpers.
- Python-only executable: simpler internally, but it bypasses the SPEC-visible `query.sh` artifact and the bead's shell-dispatch acceptance criteria.
- A generated SQL registry: unnecessary for six queries; reparsing the SQL file on startup keeps the compatibility layer reviewable.

Known subcommands should be recognized in M2.4, but their business logic remains deliberately absent. The skeleton should validate help/usage/DB/version plumbing and then exit with a clear "not implemented in this subtask" message for command bodies that belong to M2.6-M2.8. Later subtasks replace those placeholders without changing global parsing conventions.

## File Layout

Create:
- `skills/wealthfolio_query/__init__.py` — empty package marker.
- `skills/wealthfolio_query/query.py` — parser, read-only executor, schema-version check, CLI skeleton.
- `skills/wealthfolio_query/query.sh` — shell entrypoint and help surface.

No generated files belong in this task.

## `query.py` Contract

Implement these primitives with small, testable functions:

- `repo_root() -> Path`
  - Resolve from the module location, not the caller's current working directory.
- `sql_file_path() -> Path`
  - Return `repo_root() / "skills/sql/wealthfolio-queries.sql"`.
- `load_queries() -> dict[str, str]`
  - Read the SQL file as text and split on `^-- name: ` with multiline regex.
  - Preserve each query body, minus surrounding whitespace.
  - Raise a usage-style error if duplicate slugs appear.
- `load_query(slug: str) -> str`
  - Return one named query or raise a clear error for an unknown slug.
- `sql_version() -> str`
  - Read the first `-- version:` line from the SQL preamble.
- `configured_wealthfolio_version() -> str`
  - Parse `config/wealthfolio.md` for `v3.2.1` and build `20260301.1`.
  - The parser should be narrow and documented; this is a pin sanity check, not a general markdown parser.
- `validate_schema_version() -> None`
  - Compare the SQL header against the configured pin.
  - Accept either `wealthfolio v3.2.1 (build 20260301.1)` or the exact normalized string already present in the SQL header.
  - Exit 2 through the CLI wrapper on mismatch.
- `readonly_uri(db_path: str | Path) -> str`
  - Build the URI used by `sqlite3.connect(..., uri=True)`.
  - Preserve the investigation's `mode=ro&immutable=1` semantics.
  - Encode paths safely so the default macOS `Application Support` path is valid.
- `connect_readonly(db_path: str | Path) -> sqlite3.Connection`
  - Open with `mode=ro&immutable=1`.
  - Set `row_factory = sqlite3.Row`.
- `execute(slug: str, params: Mapping[str, object], db_path: str | Path | None = None) -> list[sqlite3.Row]`
  - Load the query by slug.
  - Open the DB read-only.
  - Execute using SQLite named parameters only.

Implementation note: `execute()` is allowed to run any named query that exists in the SQL file. It must never accept free-form SQL from `query.sh`. The M2.4 readonly test may call a lower-level helper with a hand-crafted write query or a temporary named slug to prove the connection itself rejects writes.

## `query.sh` Contract

Use bash with:

```bash
set -euo pipefail
```

Global behavior:
- `query.sh --help` prints the SPEC §18.1 signature block plus the M2.4 exit-code matrix.
- Unknown subcommand exits 1.
- Missing required executable dependencies (`python3`, and `sqlite3` for shell paths) exits 2 with `wealthfolio-query: <message>` on stderr.
- Missing or unreadable `WEALTHFOLIO_DB` exits 2 before any subcommand attempts to run.
- All errors are prefixed with `wealthfolio-query:`.

Known subcommands:
- `export-snapshot`
- `list-holdings`
- `get-cash-balance`
- `get-net-worth`
- `get-avg-cost`
- `get-portfolio-twr`

M2.4 may implement these as recognized placeholders after global validation. M2.6-M2.8 replace the placeholders with real command bodies. The placeholder text should name the owning future task so failures are self-explanatory during incremental development.

Shell read-only convention for later M2.6 paths:
- Use `sqlite3 -readonly "$WEALTHFOLIO_DB"` for shell-driven commands.
- Bind values only through `.parameter set`.
- Single-quote and escape `.parameter set` values explicitly before interpolation into the sqlite3 meta-command.

## Version Check

The SQL file currently opens with:

```sql
-- version: wealthfolio v3.2.1 (build 20260301.1)
```

`config/wealthfolio.md` records the same values in prose and in the §8 table. M2.4 should fail fast if the SQL header stops matching the config pin. This is the wrapper-level enforcement of SPEC invariant 14.

The comparison should intentionally be exact after normalization rather than permissive. If Wealthfolio upgrades, a future bead updates both files and the named queries together.

## Exit Codes

- `0` — help printed, or command completed successfully.
- `1` — usage error: bad flags, unknown subcommand, missing required subcommand argument.
- `2` — DB, dependency, or schema-version error.
- `3` — data-shape error reserved for later subcommands, especially positions JSON mismatches.

M2.4 should define the constants in Python and mirror them in shell comments or simple variables so later tasks do not invent new meanings.

## Task Decomposition

Implement M2.4 in this order:

2. Implement `query.py` named-query loading and version-pin validation.
3. Implement `query.py` read-only connection and parameterized `execute()`.
4. Implement `query.sh` help, dependency checks, DB-path validation, and known-subcommand dispatch placeholders.
5. Add minimal smoke coverage in M2.5 rather than placing test scripts in this bead.

## Risks

- `config/wealthfolio.md` is prose, not structured config. Keep the parser intentionally narrow and fail clearly if the expected pin lines disappear.
- The macOS default DB path contains a space. `readonly_uri()` must encode paths safely while preserving `mode=ro&immutable=1`.
- Skeleton placeholders can accidentally harden into behavior. Each placeholder should identify its owning future M2 task and be replaced by M2.6-M2.8.
- Running `bd update --claim` can touch `.beads/issues.jsonl`; plan implementation should avoid mixing tracker updates with unrelated file edits.

## Open Questions

None blocking. The only interpretation choice is whether known subcommands in M2.4 should execute partial SQL or fail as placeholders. This plan chooses placeholders because the bead explicitly says no subcommand bodies land in M2.4.
