## Approach

Use a committed SQL fixture builder, not a committed SQLite binary. `build_fixture.sh` loads the canonical schema from `research/wealthfolio-schema-v3.2.1.txt` and layers `fixture.sql` (seed INSERTs only) on top; `fixture.db` is disposable and ignored by git. This split keeps `test_schema.sh` meaningful: the fixture DB's `.schema` is diffed against the same canonical file it was built from, so any drift from a schema-dump edit fails loudly instead of being masked by a duplicated DDL copy in `fixture.sql`.

Rejected alternatives:
- Committing `fixture.db`: opaque in review and easy to let drift from the schema dump.
- Python fixture builder: more moving parts without much benefit; SQLite SQL is already the target compatibility layer.
- An anonymized real DB: risks leaking portfolio details and makes expected values harder to audit.

M2.5 is a harness task, so it should focus on reusable scaffolding and baseline invariants. Subcommand-specific assertions are added in M2.6-M2.8 using the same fixture.

## Fixture Construction

`fixture.sql` holds **seed INSERTs only**, not DDL. `build_fixture.sh` loads the canonical schema first and then the seeds:

```sh
sqlite3 fixture.db \
  ".read ../../research/wealthfolio-schema-v3.2.1.txt" \
  ".read fixture.sql"
```

Rejected alternative: embedding a copy of the schema at the top of `fixture.sql`. That creates three copies to keep in sync (schema file, fixture-embedded schema, `.schema fixture.db`) and defeats the point of `test_schema.sh`, which becomes a tautology if the fixture was built from the same verbatim paste.

Seed structure:
- Begin with a short header comment naming the anchor date and the asset/account ID conventions.
- Insert in FK-safe order: `platforms` → `accounts` → `assets` → `taxonomy_categories` → `asset_taxonomy_assignments` → `quotes` → `holdings_snapshots` → `daily_account_valuation`.
- Do not insert into `activities`; holdings-only mode is part of the contract (SPEC §5.3).

Seed data should include:
- Three active accounts:
  - INR brokerage account.
  - INR PMS account group.
  - USD brokerage account.
- One archived or inactive account with decoy rows, so active-account filters can be tested later.
- Eight active assets:
  - Five India assets covering `XNSE`, `XBOM`, and one MIC-less INR/display-code fallback.
  - Three US assets covering `XNAS`, `XNYS`, and `ARCX`.
- One India ETF, `NIFTYBEES.NS`, represented through `asset_taxonomy_assignments` and `taxonomy_categories` so `asset_type_map` can classify it as `ETF`.
- `holdings_snapshots` rows for each active account with `positions` JSON keyed by `assets.id` and values containing `quantity` and `averageCost`.
- `quotes` rows for every active asset, including enough historical rows for later TWR and quote-selection tests.
- `daily_account_valuation` rows covering at least 90 calendar days for the INR brokerage and USD brokerage accounts.
- Two cash-only accounts or cash-heavy valuation rows to support later `get-cash-balance` assertions.

Use stable synthetic IDs such as `acct_in_broker`, `asset_reliance_xnse`, and `quote_aapl_2026_04_30`. Deterministic IDs keep expected outputs readable.

## Fixture Data Conventions

Dates should be pinned around a fixed recent window rather than using `date` at build time. Suggested anchor:
- End date: `2026-04-30`.
- Ninety-day start date: `2026-01-30`.

This keeps fixture output deterministic and aligns with the current project date without introducing clock-dependent tests.

Market/ticker coverage:
- India XNSE example: `RELIANCE.NS`.
- India XBOM example: `TCS.BO` or another `.BO` synthetic symbol.
- India MIC-less example: `PMSCORE.NS` via `display_code`.
- India ETF example: `NIFTYBEES.NS`.
- US examples: `AAPL`, `BRK.B`, and an `ARCX` symbol.

The exact tickers can change during implementation if the schema imposes constraints, but the MIC coverage set from M2.3 must remain covered:

```text
XNSE, XBOM, XNAS, XNYS, ARCX, NULL
```

## Script Contracts

`build_fixture.sh`:
- `set -euo pipefail`.
- `cd` to its own directory.
- Remove only `fixture.db`, `fixture.db-wal`, `fixture.db-shm` in `tests/wealthfolio_query/`.
- Load the canonical schema from `research/wealthfolio-schema-v3.2.1.txt`, then apply `fixture.sql` seeds (see Fixture Construction above for the exact invocation).
- Exit non-zero if `sqlite3` is missing or if either `.read` step produces an error on stderr.

`run-all.sh`:
- Match the existing `tests/claim-ctl/run-all.sh` pattern.
- Run every `test_*.sh` except helper files.
- Print a concise pass count.
- Exit non-zero if any subtest fails.

`test_readonly.sh`:
- Rebuild the fixture first or assert `fixture.db` exists after `build_fixture.sh`.
- Run `sqlite3 -readonly fixture.db 'UPDATE accounts SET name = "x"'`.
- Assert non-zero exit.
- Assert stderr contains `readonly`.
- Assert `skills/sql/wealthfolio-queries.sql` has no top-level DDL/DML. The SQL file is heavily commented (preamble and provenance blocks mention `CREATE`, `UPDATE`, `INSERT` in prose), so the grep MUST strip `^\s*--` lines before matching. Use:

```sh
sed -E 's|--.*$||' skills/sql/wealthfolio-queries.sql \
  | grep -cE '^\s*(CREATE|INSERT|UPDATE|DELETE|DROP|ALTER)\b'
```

The assertion is that the count is `0`. A simpler `grep -c` without the comment strip will false-positive on the preamble and must not be used.

`test_schema.sh`:
- Rebuild the fixture.
- Compare `sqlite3 fixture.db '.schema'` to `research/wealthfolio-schema-v3.2.1.txt`.
- Normalize harmless trailing whitespace before diffing.
- Do not hide real DDL differences.
- Also assert the MIC coverage invariant from M2.3 (investigation §4):

```sh
sqlite3 -readonly fixture.db \
  "SELECT DISTINCT COALESCE(instrument_exchange_mic, '<NULL>')
   FROM assets WHERE is_active = 1 ORDER BY 1"
```

The expected sorted set is `<NULL>, ARCX, XBOM, XNAS, XNSE, XNYS`. If the seed data or real Wealthfolio data drifts to an unexpected MIC, the test fails loudly. This is a fixture-side check, not a live-DB check; M2.12 owns verification against the user's real DB.

`test_wrapper_skeleton.sh`:
- Assert `skills/wealthfolio_query/query.sh --help` exits 0 and includes all six subcommand names.
- Assert an unknown subcommand exits 1.
- Assert `WEALTHFOLIO_DB=` with a known subcommand exits 2 and includes `wealthfolio-query:` on stderr.
- Assert Python `load_queries()` returns all six named slugs. The test must run Python from the repo root so the `skills.wealthfolio_query.query` import resolves without a PYTHONPATH hack. Example:

```sh
( cd "$REPO_ROOT" && python3 -c '
from skills.wealthfolio_query.query import load_queries
slugs = set(load_queries())
assert {"export-snapshot","list-holdings","get-cash-balance",
        "get-net-worth","get-avg-cost","get-portfolio-twr"} <= slugs, slugs
' )
```

- Assert the schema-version check passes against the current SQL/config pair by invoking the M2.4 `validate_schema_version()` helper via the same repo-root `python3 -c` pattern.

The fixture builder may remove `fixture.db`, `fixture.db-wal`, and `fixture.db-shm` inside `tests/wealthfolio_query/` only. It must not delete anything outside that directory.

## Risks

- `.schema` output ordering or whitespace may differ from the committed schema dump. Normalize whitespace only; do not normalize away DDL order or column differences.
- The fixture DDL is a copied schema. When Wealthfolio upgrades, the schema dump and fixture must move together under invariant 14.
- Seed rows can accidentally depend on future subcommand behavior. Keep M2.5 assertions focused on harness invariants; leave business values to M2.6-M2.8.
- The SQL static DDL/DML grep false-positives on comments today — the current `skills/sql/wealthfolio-queries.sql` preamble mentions `CREATE`/`UPDATE`/`INSERT` in prose in multiple places. The Script Contracts section above pins the comment-stripping `sed`+`grep` combination; do not regress to a bare `grep -cE`.
- There is no existing Makefile. Creating a minimal one is acceptable for this bead, but it should not become a broader build-system redesign.
