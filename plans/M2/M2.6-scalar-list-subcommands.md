## Approach

Keep dispatch in shell. All four subcommands are "run SQL → format one row-set → print": they do not need Python post-processing. `query.sh` builds a `.parameter set` preamble, runs `sqlite3 -readonly "$WEALTHFOLIO_DB"` with the named-query body loaded via a small helper, and formats output.

Reject alternatives:
- Route these four through `python3 -m skills.wealthfolio_query.query <subcmd>`: would duplicate `query.py`'s logic for no gain. Shell is the SPEC §17-named surface and `sqlite3 -readonly` is well-suited to these one-shot reads.
- Embed SQL bodies inline in the shell script: breaks the single-compatibility-layer invariant (§19.2 invariant 14). Always load from `skills/sql/wealthfolio-queries.sql`.

The shared helper (a bash function, not a new file) extracts one named-query body by slug from the SQL file on demand and writes it to a temp file that `sqlite3` reads via `.read`. This keeps the parameter-binding flow (`.parameter set :name 'value'` preamble + `.read /tmp/slug.sql`) simple and avoids shell-escaping SQL bodies.

### Parameter-binding safety

Every `.parameter set` value is single-quoted and any embedded `'` is escaped to `''`. A small bash helper does this escape. User-supplied values (`--scope-value`, `--currency`, `--ticker`, `--date`) never reach a SQL body as text — only as bound parameters.

### `list-holdings` SQL widening

The existing `list-holdings` body in `skills/sql/wealthfolio-queries.sql` returns 7 columns. M2.6 replaces it with a 13-column shape that mirrors `export-snapshot` exactly (minus the thesis join, which lives in `export_snapshot.py`). The new column list:

```
ticker, name, market, currency, account, account_group, asset_type,
quantity, avg_cost, snapshot_price, snapshot_market_value,
allocation_pct, unrealized_pl_pct
```

Reuses the `asset_type_map` CTE from `export-snapshot`, the Yahoo-ticker CASE, the MIC→market CASE with `accounts.currency` fallback, and the post-scope `allocation_pct = market_value / SUM(market_value) OVER ()`. `--market all` passes the market filter; `--scope-type/--scope-value` is still filtered in the outer `WHERE`. Query ends with `ORDER BY ticker, account` for byte-identical re-runs (investigation §10, §12).

Header comment update: replace the `returns:` line listing the 7 narrow columns with the 13-column list and note that this shape is deliberately aligned with `export-snapshot`.

### `get-cash-balance` SQL change

Current SQL requires `:currency` and filters `acc.currency = :currency`. Widen to support the `all`/omitted mode:

- Parameter `:currency` always bound; the wrapper binds the literal string `'all'` when the flag is omitted or explicitly `all`.
- SQL `WHERE` becomes:
  - If `:currency = 'all'`: no currency filter, sum `CAST(cash_balance AS REAL) * CAST(fx_rate_to_base AS REAL)`.
  - Else: `acc.currency = :currency`, sum `CAST(cash_balance AS REAL)` (native, no FX factor).
- Single query covers both with a CASE in the `SUM(...)` and a `(:currency = 'all' OR acc.currency = :currency)` in the `WHERE`.

Scope filter block stays as-is (market/account_group/account tri-mode, currently keyed to the INR↔india / USD↔us bijection for `market`).

### `get-net-worth` SQL change

Current SQL has no market filter and no currency logic. Widen:

- Parameters: `:date` (nullable), `:market` (`india|us|all`), `:currency` (`INR|USD|'none'` where `'none'` means "emit in base").
- Filter accounts by market via the bijection: `(:market = 'all' OR (:market = 'india' AND acc.currency = 'INR') OR (:market = 'us' AND acc.currency = 'USD'))`.
- Emit two SUMs: `base_total = SUM(total_value * fx_rate_to_base)` and a picked `fx_day = MAX(CASE WHEN acc.currency = :currency THEN fx_rate_to_base END)` on the selected date. The shell-side formatter then computes `base_total / fx_day` when `:currency != 'none'`, else emits `base_total` directly. SQL returns both columns; shell does the division.

### `get-avg-cost` zero-row handling

Existing SQL returns `NULL` from `NULLIF(SUM(quantity), 0)` when no position matches. The wrapper inspects the one-row result: if the value is empty/NULL, exit 2 with `wealthfolio-query: ticker not in active holdings: <ticker>` on stderr. No SQL change required.

### Output format rules (from investigation §9.1, §11)

- `list-holdings`: `--format json` (default) | `--format csv`. JSON is an array of objects with the 13 columns. CSV has the 13-column header line, rows `ORDER BY ticker, account`.
- `get-cash-balance`, `get-net-worth`, `get-avg-cost`: bare number on stdout, single trailing newline. No JSON wrapper. Python `float` format via `printf '%s\n'` where the value already comes out of `sqlite3` as a decimal string; do not re-format or round.
- Empty scope on `get-cash-balance`: stdout `0.0\n`, exit 0 (silent — `--strict-scope` wiring lands in M2.9).
- Empty market/scope on `get-net-worth`: stdout `0.0\n`, exit 0.

## `query.sh` Contract Additions

After the M2.4 globals and validation:

```bash
_load_named_query() {
  # $1 = slug; prints the body of -- name: $1 from skills/sql/wealthfolio-queries.sql
  python3 -c "import sys
from skills.wealthfolio_query.query import load_query
sys.stdout.write(load_query('$1'))"
}

_esc_sq() {
  # Escape single quotes for .parameter set values
  printf "%s" "$1" | sed "s/'/''/g"
}

_run_named_query() {
  # $1 = slug, remaining args: alternating name value pairs
  local slug="$1"; shift
  local tmp_sql; tmp_sql=$(mktemp)
  trap 'rm -f "$tmp_sql"' RETURN
  _load_named_query "$slug" > "$tmp_sql"
  {
    echo ".mode list"
    echo ".separator |"
    while [[ $# -gt 0 ]]; do
      local name="$1"; local value="$2"; shift 2
      printf ".parameter set :%s '%s'\n" "$name" "$(_esc_sq "$value")"
    done
    echo ".read $tmp_sql"
  } | sqlite3 -readonly "$WEALTHFOLIO_DB"
}
```

Each subcommand function:
- Parses its own flags (`getopts`-style long-flag loop; unknown flag → `exit 1`).
- Calls `_run_named_query <slug> <name1> <value1> ...`.
- Formats stdout (JSON for `list-holdings --format json`, CSV for `--format csv`, bare number for the three scalars).

For `list-holdings`, default `--format` is JSON. The SQL output via `.mode list` `.separator |` gives piped rows the formatter turns into either JSON (via `python3 -c` one-liner that reads stdin and dumps a list of dicts) or CSV (via the same helper emitting `csv.writer` output). This keeps the scalar path pure shell and isolates the JSON/CSV branching to `list-holdings`.

Rejected alternative: use `sqlite3 -json`. It works, but combining `-json` with `.parameter set` and `.read` in a pipeline is fragile across sqlite3 versions, and CSV still needs a separate mode. One formatting helper keeps behavior uniform.

### Subcommand-specific flag handling

**`list-holdings`** (per SPEC §18.1 + investigation §9.1):
- `--market india|us|all` (default `all`).
- `--scope-type market|account_group|account` (optional; absence = no scope filter).
- `--scope-value <value>` (required iff `--scope-type` given; missing pair → exit 1).
- `--format json|csv` (default `json`).

When `--scope-type` is absent, bind `:scope_type = 'market'` and `:scope_value = 'all'`, and update the SQL `WHERE` to treat `:scope_type = 'market' AND :scope_value = 'all'` as "pass everything". (This is a small SQL-side adjustment inside the already-existing tri-mode block.)

**`get-cash-balance`**:
- `--currency INR|USD|all` (default `all`).
- `--scope-type`, `--scope-value` — same rules as `list-holdings`.

**`get-net-worth`**:
- `--market india|us|all` (default `all`).
- `--currency INR|USD` (optional; omitted = base-currency output).
- `--date <YYYY-MM-DD>` (optional; defaults to latest `valuation_date` across scoped accounts).

**`get-avg-cost`**:
- Single positional argument `<ticker>`. No flags.
- Missing argument → exit 1 with `wealthfolio-query: get-avg-cost requires a ticker argument`.
- Result NULL/empty → exit 2 with `wealthfolio-query: ticker not in active holdings: <ticker>`.

## Exit codes

Matches M2.4:
- `0` — success, even when the result is empty (cash-balance on empty scope, list-holdings on filtered-out slice).
- `1` — usage error: unknown flag, missing required flag pair, missing positional arg.
- `2` — DB / schema / data-not-found:
  - `get-avg-cost` on a non-held ticker exits 2 (genuine data miss).
  - DB-open failures, schema-version mismatch from M2.4.
- `3` — reserved for later subcommands (data-shape mismatches).

`--strict-scope` bad-scope exit 2 for `list-holdings` / `get-cash-balance` lands in M2.9, not here.

## Risks

- **SQL-side currency/market branching via bound parameters.** SQLite accepts the `(:flag = 'all' OR ...)` pattern trivially, but the `MAX(CASE WHEN acc.currency = :currency ...)` in `get-net-worth` is less common. If the fixture reveals sqlite3 quirks (e.g. `CASE` inside `MAX` returning `NULL` because no row matched on the selected date), fall back to a two-pass approach: query base total separately and pick the FX factor from a second small query. Decide during implementation.
- **`.parameter set` + `.read` interaction.** Older sqlite3 builds accept `.parameter set` but do not persist values into `.read`'d files. The M2.5 container image's sqlite3 version is the one that matters. Verify with a quick smoke test in the M2.4 RO-open path before hardening the helper; if it breaks, switch to a `python3 -m skills.wealthfolio_query.query` dispatch for all four subcommands (adds a Python hop but keeps parameter binding first-class).
- **Fixture expected values.** Hand-computed base-currency sums depend on the fixture's `fx_rate_to_base` (83.2) and the `valuation_seed` step math. Compute expected values inside each test via a short Python snippet that mirrors the fixture arithmetic, rather than hardcoding opaque constants — makes the assertions self-auditable if fixture seeds ever change.
- **`list-holdings` fixture row count.** The fixture seeds multi-account positions (Reliance in both `acct_in_broker` and `acct_in_pms`). The exact row count and per-row `allocation_pct` must be verified against the fixture during implementation; update the test accordingly rather than locking in a number upfront.
