## Approach

All math in Python. SQL owns only the per-day base-currency aggregation; Python owns differencing, forward-fill, chaining, and JSON emission. Rationale from investigation §5:

- SQLite recursive-CTE chaining against `TEXT`-stored decimals is fragile.
- Forward-fill requires a calendar generator that is cleaner in Python.
- JSON output with a `series` array maps naturally to a list comprehension.

### Endpoint convention (investigation §5)

`--start :s --end :e` means *"TWR from close of `:s` to close of `:e`"*:

- Daily returns produced: `r[s+1], r[s+2], …, r[e]` — total `(e − s)` returns.
- `V[s]` is the anchor; `r[s]` is not in the product.
- `--start = --end` → zero-day window → `TWR = 0.0`, empty `series`.
- Caller-side semantics: M6's "1m window ending today" → `--start = today − 30d`, `--end = today`. No off-by-one.

### Differencing rule

From the SQL we get `cum[t]` per day. Daily contribution:

```
C[t] = cum[t] - cum[t-1]
```

Where `t-1` is the previous row in date-sorted order (after forward-fill). `C[s]` is unused (because `r[s]` is not in the product). If `cum[s-1]` is not available (series starts at `:s`), we have no choice — `C[s+1] = cum[s+1] - cum[s]`, which is correct.

### Forward-fill rule

SQL returns only dates where at least one scoped account has a valuation row. Python:

1. Build the full calendar from `:start` to `:end` inclusive via `date` arithmetic.
2. For each calendar day:
   - If the day is present in the SQL result → use `(V, cum)` from that row.
   - Else → forward-fill from the previous day: `V[t] = V[t-1]`, `cum[t] = cum[t-1]`, which yields `C[t] = 0` and `r[t] = 0`, a no-op factor of `1` in the product.
3. If `:start` itself has no row, look backwards at the SQL result for the latest row on or before `:start` and use its values as `V[s]`, `cum[s]`. If no such row exists, exit 3 (data-shape error) with `wealthfolio-query: no valuation rows at or before :start` — we cannot chain from nothing.

Why forward-fill over "skip the day": mathematically equivalent (both produce a factor of 1), but the explicit fill makes the `series` output dense and debuggable. M6's benchmark assembler only consumes `twr`; the dense series is for `test_twr.sh` + human inspection.

### Chained product

```python
twr = 1.0
for row in series_after_anchor:
    twr *= (1.0 + row.r)
twr -= 1.0
```

`CAST(... AS REAL)` gives ~15 decimal digits of precision; Python `float` preserves that. 1bp tolerance in `test_twr.sh` is orders of magnitude looser than numeric noise here.

### JSON output

```json
{
  "twr": 0.0421,
  "start": "2026-03-30",
  "end": "2026-04-30",
  "market": "india",
  "series": [
    { "date": "2026-03-30", "V": 735388.89, "C": 0.0, "r": null },
    { "date": "2026-03-31", "V": 735827.78, "C": 222.22, "r": 0.000294 },
    ...
  ]
}
```

- `series[0]` is the anchor: `r: null` (not in the product), `C: 0.0` (no prior day to diff).
- Subsequent entries carry computed `r` and `C`.
- `V` and `C` are Python floats; do not round — preserve what the SQL `CAST AS REAL` produced.
- `twr` is rounded to 6 decimals for display cleanliness; the test tolerance (1bp = 1e-4) is well within that.
- `market` added for debugging; M6 does not require it but it's cheap and makes the JSON self-describing.

Default destination: stdout. No `--output` flag in the SPEC §18.1 signature — M6 pipes stdout into `jq`.

## `portfolio_twr.py` Contract

```
python3 -m skills.wealthfolio_query.portfolio_twr \
  --market india|us \
  --start YYYY-MM-DD \
  --end YYYY-MM-DD
```

- `--market`, `--start`, `--end` all required. Missing → exit 1.
- `:end >= :start` enforced; `:end < :start` → exit 1 with `wealthfolio-query: --end must be >= --start`.
- `:start == :end` → emit `{"twr": 0.0, "start": ..., "end": ..., "market": ..., "series": []}` and exit 0.
- DB path from `WEALTHFOLIO_DB`. RO-open via `query.connect_readonly()`.
- Exit codes:
  - 0 success.
  - 1 usage error.
  - 2 DB / schema-version (inherited from M2.4 globals).
  - 3 data-shape: no valuation row at or before `:start` (cannot chain).

Internally the module imports from `skills.wealthfolio_query.query`:
- `load_query("get-portfolio-twr")` implicitly via `execute()`.
- `execute(slug, params, db_path=None)` returns `list[sqlite3.Row]` — one row per `valuation_date` with `(date, portfolio_value_base, cumulative_net_contribution_base)`.

Note: the SQL currently filters `WHERE dav.valuation_date BETWEEN :start AND :end`. For the "no row on `:start`, need a prior-day anchor" case we need one row before `:start`. Two options, decided at implementation time:

- (a) Extend the SQL `WHERE` to `BETWEEN date(:start, '-14 days') AND :end`, then the Python side picks the latest row `<= :start` as the anchor.
- (b) Run a second `execute()` for the anchor: `SELECT ... WHERE valuation_date <= :start ORDER BY valuation_date DESC LIMIT 1`. Requires an additional named query (`get-portfolio-twr-anchor`) in `skills/sql/wealthfolio-queries.sql`.

Prefer (a) — one query, simpler. The 14-day lookback window is arbitrary but safely covers weekends and long holidays; if real-world data has gaps longer than 14 days around `:start`, the anchor-not-found error fires and the caller adjusts the window. Record the 14-day constant at the top of `portfolio_twr.py` with a comment citing this plan.

3. **Forward-fill gap behavior.**
   - If `acct_in_broker` has a dense series (current fixture: every day 2026-01-30..04-30), temporarily narrow the test to a range where one account has a gap.    - `--market us --start 2026-03-30 --end 2026-04-30` → non-NaN `twr`, `len(series) == 32`, base-currency V reflects `fx_rate_to_base = 83.2`.

5. **Usage errors.**
   - Missing `--start` → exit 1.

   - `--market europe` → exit 1 (SPEC-constrained).

7. **No anchor.**
   - `--start 2020-01-01 --end 2020-01-31` (before any fixture data) → exit 3 with `no valuation rows at or before :start`.

## Risks

- **Anchor-row lookback window.** 14 days covers weekends/holidays but not long vacation gaps or discontinued reporting. If M6 surfaces a real gap, either widen the window constant or require users to pass `--start` on a day with coverage. Document the constant; do not over-engineer.
- **`cumulative_net_contribution` interpretation.** The v3.2.1 column is `net_contribution` and upstream re-emits the cumulative running total each row (pinned in M2.3 + investigation §5). If a future minor version switches to per-day deltas, the differencing logic is wrong and all TWR values become garbage. Detection: the schema-version check fires on any header mismatch, and a regression test that asserts a known `C[t]` value against the fixture will flag it before the TWR assertion does. Add that `C[t]` assertion explicitly in `test_twr.sh`.
- **Float drift across rebuilds.** Not a concern for TWR (ratio math, 1bp tolerance), but the `series` JSON values may differ by ulp-level noise across runs. Byte-identical `series` is not required; only `twr` has a tolerance assertion.
- **Empty scope at :end.** If `:end` is after all fixture data (e.g. `2026-05-01` against the 2026-04-30 cap), forward-fill carries V/cum to `:end`, producing zero-return days. This is arguably correct (the portfolio did not change), but worth an explicit test that `--end` past the last valuation day does not exit non-zero. Add as assertion #8 if implementation time permits; otherwise, document the behavior in the module docstring and leave testing to M6.
- **Fixture gap-seed side effects.** If M2.8 adds gap deletes to `fixture.sql`, the M2.5 `test_schema.sh` diff is unaffected (schema unchanged) but `test_wrapper_skeleton.sh` and any M2.6 tests that hardcode per-account valuation counts need re-verification. Prefer inserting gap coverage via a *different account* (or a new sparse account) over deleting rows from existing accounts to minimize M2.6 test churn.
