Findings:
- `Position` is annotated with `#[serde(rename_all = "camelCase")]`.
- The fields needed by M2 are `asset_id`, `quantity`, `average_cost`, `total_cost_basis`, and `currency`; their JSON names are `assetId`, `quantity`, `averageCost`, `totalCostBasis`, and `currency`.
- `AccountStateSnapshot.positions` is declared as `HashMap<String, Position>` with an upstream comment saying the map is `asset_id -> Position`.
- `HoldingsCalculator::get_or_create_position_mut_cached` inserts positions with `.entry(asset_id.to_string())`, confirming the JSON object key is the Wealthfolio `assets.id`, not `instrument_key` or `display_code`.
- `AccountStateSnapshotDB::from(AccountStateSnapshot)` persists `positions` with `serde_json::to_string(&domain.positions)`, so the DB column is a direct JSON object keyed by `assets.id`.
- Workspace `Cargo.toml` configures `rust_decimal = { version = "1.39", features = ["maths", "serde-float"] }`; SQLite `json_extract(... AS REAL)` remains compatible with the serialized Decimal values.

## Asset Type Mapping

`assets.instrument_type = 'EQUITY'` is not enough to distinguish stocks from ETFs because upstream treats stocks, ETFs, and funds as the same provider-routing enum. The v3.2.1 taxonomy migration seeds an `instrument_type` taxonomy with stock-like categories under `EQUITY_SECURITY` and ETF-like categories under `ETP`.

For `export-snapshot`, add an `asset_type_map` CTE:

```sql
asset_type_map AS (
    SELECT
        ata.asset_id,
        MAX(CASE
              WHEN tc.id = 'ETF' OR tc.parent_id = 'ETP' THEN 1
              ELSE 0
            END) AS is_etf
    FROM asset_taxonomy_assignments ata
    JOIN taxonomy_categories tc
      ON tc.taxonomy_id = ata.taxonomy_id
     AND tc.id = ata.category_id
    WHERE ata.taxonomy_id = 'instrument_type'
    GROUP BY ata.asset_id
)
```

Then map:

```sql
CASE
  WHEN COALESCE(atm.is_etf, 0) = 1 THEN 'ETF'
  ELSE 'stock'
END AS asset_type
```

This keeps the SPEC §7.1.1 two-value contract (`stock` or `ETF`) while allowing the fixture to model `NIFTYBEES.NS` as an ETF through taxonomy assignment. Missing taxonomy assignment falls back to `stock`, which is acceptable for the India-first MVP and will be made visible by the fixture tests.

## MIC Enumeration Handoff

M2.3 records the expected MIC set currently embedded in the market/ticker CASE expressions:

- India: `XNSE`, `XBOM`
- US: `XNAS`, `XNYS`, `ARCX`

M2.5 should add a test-harness query:

```sql
SELECT DISTINCT instrument_exchange_mic
FROM assets
WHERE instrument_exchange_mic IS NOT NULL
ORDER BY instrument_exchange_mic;
```

The fixture assertion should fail if a fixture asset introduces an unhandled MIC. Broader real-world MIC expansion, such as `XASE` or `BATS`, is a later schema/query review rather than an implicit M2.3 change.

The M2.5 fixture should seed an ETF taxonomy assignment to lock the intended path, and live-data verification in M2.12 should report any unclassified ETF-like holdings.
- MIC drift: the current SQL only recognizes `XNSE`, `XBOM`, `XNAS`, `XNYS`, and `ARCX`. M2.5 catches fixture drift; real-data drift requires a schema/query review under invariant 14.
