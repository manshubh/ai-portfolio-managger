## Approach

Keep SQL → 13 columns (no thesis); join thesis in Python. This is the investigation's decision: CSV escaping of free-text thesis strings is first-class in `csv.writer`, and the thesis source-of-truth is a YAML file outside the DB. Attempting the merge inside SQL would require pre-loading YAML into a transient table, which complicates the read-only contract.

Execution order inside `export_snapshot.py`:

1. Parse flags (argparse).
2. Resolve `--theses` from `--market` if absent: `input/{market}/theses.yaml`. Missing file → one-line stderr warning, empty dict continues.
3. Call `skills.wealthfolio_query.query.execute("export-snapshot", params, db_path)` → list of `sqlite3.Row`.
4. Load thesis dict: `yaml.safe_load(theses_path).get("theses", {})`. Any non-string values → stringify and warn; empty string for missing key.
5. For each row, build the 14-tuple (13 SQL columns + thesis lookup by `row["ticker"]`).
6. Stream to `--output` path (or stdout) via `csv.writer(quoting=csv.QUOTE_MINIMAL)`. Header line matches SPEC §7.1.1 byte-for-byte.
7. When `--output` is set, the wrapper exits silently (SPEC §6.8 narrowing: we never echo preview rows).

Rejected alternatives:
- Have `query.sh` do the SQL and pipe into `export_snapshot.py`: fine, but two processes with different env requirements. A single `python3 -m skills.wealthfolio_query.export_snapshot` call is simpler and `execute()` in `query.py` already owns the RO-open.
- Merge theses as a left join to a CTE populated via `VALUES` inside the SQL named query: breaks the compatibility-layer contract — the SQL file becomes YAML-aware.

### Theses-YAML contract

Loader behavior in `export_snapshot.py`:

- File missing → stderr warning `wealthfolio-query: theses file not found at {path}; continuing with empty thesis column`, exit continues (SPEC §19.2 invariant 13).
- File exists but top-level `theses:` key missing → treat as empty dict. Warn once.
- File exists with malformed YAML → exit 3 (data-shape error) with `wealthfolio-query: theses file is not valid YAML: {path}: {err}`.
- Values that are not strings (numbers, lists) → coerce via `str(...)` and warn once per occurrence.
- Key lookup is exact-match on canonical Yahoo ticker. No fallback, no ADR mapping (out of scope per investigation §14).

### CSV contract

Header line, exact:

```
ticker,name,market,currency,account,account_group,asset_type,quantity,avg_cost,snapshot_price,snapshot_market_value,allocation_pct,unrealized_pl_pct,thesis
```

`csv.writer` with:
- `quoting=csv.QUOTE_MINIMAL`
- `lineterminator="\n"` — explicit so we do not emit `\r\n` on accident (Python default on Windows; not an issue on darwin but fixing it keeps the byte-identical assertion portable).
- Numeric columns (`quantity`, `avg_cost`, `snapshot_price`, `snapshot_market_value`, `allocation_pct`, `unrealized_pl_pct`) emit whatever `CAST(... AS REAL)` produced via `sqlite3` as-is. Do not re-format or round in Python — the SQL value is the authority, and byte-identical re-runs require matching representations.
- `thesis` column escapes commas, quotes, and newlines automatically.

## `export_snapshot.py` Contract

```
python3 -m skills.wealthfolio_query.export_snapshot \
  --market india|us \
  --scope-type market|account_group|account \
  --scope-value <value> \
  [--theses <path>] \
  [--output <path>]
```

- `--market`, `--scope-type`, `--scope-value` are required. Missing → exit 1 with a clear usage message.
- `--theses` defaults to `input/{market}/theses.yaml` under `repo_root()`.
- `--output` omitted → write CSV to stdout. Set → write to file, silent stdout, non-zero exit 2 on IO error.
- DB path from `WEALTHFOLIO_DB`.
- Exit codes match the M2.4 matrix: 0 success, 1 usage, 2 DB/IO, 3 theses-YAML malformed.

## Risks

- **Numeric stringification drift.** If `sqlite3.Row` values arrive as Python `float`, `csv.writer` stringifies via `repr(float)` which can emit `2377.7777777777778` vs the SQL text `2377.78`. To preserve byte-identical re-runs, fetch rows as text: either add `conn.text_factory = str` *and* keep SQL `CAST(... AS REAL)` outputs routed through `print(row[i])` directly, or query with explicit `printf('%.6f', ...)` wrappers in SQL. Decide during implementation; the assertion target is that two consecutive runs produce byte-identical files, which is tractable under either approach.
- **PyYAML availability.** If the skill image lacks PyYAML, choose between adding it to the image (clean) or hand-parsing the flat `theses:` block (narrow surface, 20-line function). Prefer the image addition; fallback ready if it bloats the image.
- **Fixture row count assumptions.** The exact India/US row counts depend on the fixture's `holdings_snapshots.positions` JSON. The numbers in §`test_export_snapshot.sh` above are derived from a read of `fixture.sql` but must be re-verified during implementation before hardcoding in assertions.
- **Real `input/india/theses.yaml` has 52 tickers** but the fixture only models 5 India assets. Running `--market india` against the real sidecar path in the assertion (step 7) means most theses entries go unused — harmless since the CSV only contains fixture rows, but the test must not assert absolute row count against the sidecar's ticker count.
- **`--output` path creation.** If the directory does not exist, `open(path, "w")` fails. Exit 2 with a clear message; do not auto-`mkdir -p` (callers own their output directory).
