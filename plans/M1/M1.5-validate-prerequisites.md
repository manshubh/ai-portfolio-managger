## Scope

In scope — one executable script at [skills/scripts/validate-prerequisites.sh](../../skills/scripts/validate-prerequisites.sh) that pre-checks every `temp/research/stocks/{TICKER}.md` for phase-specific prerequisites:

| Phase arg | Prerequisite | Source |
|---|---|---|
| `4` | every stock file has both `[Fund]` and `[Valu]` tag-header lines | SPEC §7.3, §10.5 |
| `5` | every stock file has a `## Scoring` header | SPEC §10.5 / §10.6 (Phase 5 reads scoring) |

Exit non-zero (1) on any missing prerequisite; print `<TICKER>: missing <pattern>` lines to stderr. Exit non-zero (2) on usage errors (bad/missing phase arg, missing stocks dir, no stock files).

This script is **standalone** — it does not source [`_claim-common.sh`](../../skills/scripts/_claim-common.sh) (investigation §3). It's a grep-and-existence check, not a claim-ctl operation.

Out of scope:

- Phase 2/4/5 claim shims — M1.2/M1.3/M1.4 (shipped).
- Concurrent / stale tests — M1.6 / M1.7.
- Tests for this script — M1.8 owns prerequisite-check coverage.
- [skills/scripts/README.md](../../skills/scripts/README.md) entry for `validate-prerequisites` — M1.9.
- `shellcheck skills/scripts/*.sh` epic-level sweep — M1.10. Local lint in this subtask.
- Body validity for `[Fund]`/`[Valu]`/`## Scoring` (specific metrics present, scoring rationale well-formed). Header presence only — body validation is stock-linter / scoring-engine territory (M9, M10).
- Wiring the call into Phase 4 / Phase 5 prompts. The Phase prompts in M9 / M10 / M11 are responsible for invoking this script before any `claim-*`.

## Approach

### File shape

```bash
#!/usr/bin/env bash
# validate-prerequisites.sh — pre-Phase 4/5 stock-file gate. SPEC §18.8.
# Usage: validate-prerequisites.sh <phase-number>
#   phase=4 → every temp/research/stocks/*.md must contain [Fund] AND [Valu]
#   phase=5 → every temp/research/stocks/*.md must contain ## Scoring header
# Exit 1 on missing prerequisites (offending tickers on stderr).
# Exit 2 on usage errors (bad phase arg, missing stocks dir, no stock files).
set -euo pipefail

usage() {
  printf 'usage: %s <phase-number>  (valid: 4, 5)\n' "$0" >&2
  exit 2
}

[[ $# -eq 1 ]] || usage
phase="$1"

REPO_ROOT="${CLAIM_CTL_ROOT:-$(cd "$(dirname "${BASH_SOURCE[0]}")/../.." && pwd)}"
STOCKS_DIR="${REPO_ROOT}/temp/research/stocks"

case "$phase" in
  4) patterns=('^\[Fund\]' '^\[Valu\]') ;;
  5) patterns=('^## Scoring[[:space:]]*$') ;;
  *) usage ;;
esac

if [[ ! -d "$STOCKS_DIR" ]]; then
  printf 'validate-prerequisites: stocks dir does not exist: %s\n' "$STOCKS_DIR" >&2
  exit 2
fi

failed=0
found=0
while IFS= read -r f; do
  found=1
  ticker="$(basename "$f" .md)"
  for pat in "${patterns[@]}"; do
    if ! grep -qE "$pat" "$f"; then
      printf '%s: missing %s\n' "$ticker" "$pat" >&2
      failed=1
    fi
  done
done < <(find "$STOCKS_DIR" -mindepth 1 -maxdepth 1 -type f -name '*.md' | sort)

if [[ "$found" -eq 0 ]]; then
  printf 'validate-prerequisites: no stock files in %s\n' "$STOCKS_DIR" >&2
  exit 2
fi

exit "$failed"
```

> **Implementation deviation (recorded per CLAUDE.md §5):** initial draft used `mapfile -t files < <(find ...)`. macOS host bash is 3.2 (no `mapfile` builtin), so the loop was switched to `while IFS= read -r f; do …; done < <(find ... | sort)` — same pattern [`_claim-common.sh:94`](../../skills/scripts/_claim-common.sh#L94) uses, with a `found=0/1` flag replacing `${#files[@]} -eq 0` for the empty-dir check. Behavior identical (deterministic sort, single failure surface, all-files iteration).

### Design choices (locked in this plan)

1. **Empty / missing `temp/research/stocks/`** — exit 2 with a stderr message. A vacuous "all files pass" on an empty dir would silently let the Phase 4/5 caller proceed to a no-op claim batch and produce a confusing "0 tickers" downstream. Loud failure beats silent success here. (Decision per session conversation.)
2. **Header-only checks**, regex-anchored at column 0:
   - Phase 4: `^\[Fund\]` and `^\[Valu\]` — matches the dense one-line-per-tag layout in [SPEC §7.3](../../docs/SPEC.md#L410). Continuation lines starting with `|` are not validated; that's body content.
   - Phase 5: `^## Scoring[[:space:]]*$` — identical regex to [complete-scoring.sh](../../skills/scripts/complete-scoring.sh). Trailing whitespace tolerated; trailing content rejected.
3. **Phase-number whitelist.** Only `4` and `5` accepted; any other value (including `2`, `3`, `6`, `7`) hits the usage error. SPEC §18.8 documents only these two; phases 1–3 / 6–7 have no prerequisite check.
4. **Exit code shape.** `1` = data failure (one or more tickers failed validation). `2` = usage failure (bad arg, no stocks dir, empty stocks dir). Lets callers distinguish "your data is bad" from "you called this wrong" — common bash convention.
5. **Stderr format.** One line per `(ticker, pattern)` miss: `INFY.NS: missing ^\[Fund\]`. Iterate all files and all patterns before exiting so the caller sees every failure in one shot, not just the first.
6. **`REPO_ROOT` resolution mirrors `_claim-common.sh`.** Resolve from `${BASH_SOURCE[0]}/../..` with a `CLAIM_CTL_ROOT` env override for test isolation (matches [`_claim-common.sh:17`](../../skills/scripts/_claim-common.sh#L17)). Even though we don't source the library, using the same env-var name keeps test-fixture authoring consistent across all of M1.
7. **`set -euo pipefail`** for parity with the rest of `skills/scripts/`.
8. **Deterministic order.** `find ... | sort` so failure output is reproducible across runs.

### Why no shared library

Investigation §3 is explicit: "`validate-prerequisites.sh` — separate script. It is a grep + existence check over stock files, not a claim-ctl operation, and does not source `_claim-common.sh`." The library's API surface (`_claim_batch`, `_claim_complete`, `_claim_check_progress`, `_claim_reclaim_stale`) is about per-ticker dirs under a `CLAIM_ROOT`. This script reads stock-file content under `temp/research/stocks/`, which the library never touches. Sourcing it would couple two unrelated abstractions.

## Rejected alternatives

| Alternative | Why rejected |
|---|---|
| Exit 0 on empty `temp/research/stocks/` (vacuously true) | Silent pass into Phase 4/5 with nothing to score is a worse failure mode than a loud "no stock files found" error. (User decision in session.) |
| Match `[Fund]` / `[Valu]` anywhere on the line (not anchored) | Could match `[Fund]` mentioned inside `[News]` or `[Phil]` body content. SPEC §7.3 places these at column 0 as tag headers; `^\[Fund\]` is the strict + correct check. |
| Validate body content (specific metrics in `[Fund]`, score format in `## Scoring`) | Out of scope per investigation §6.5/§6.6 — header-only is the contract; body validation is the stock-linter (M9) and scoring-engine (M10) concern. Coupling M1 to body content also blocks shipping until those land. |
| Source `_claim-common.sh` to reuse `REPO_ROOT` | Investigation §3 explicitly forbids this. Re-deriving `REPO_ROOT` is one line; the cost of an extra tiny duplication is much less than the cost of a wrong abstraction. |
| Accept `<phase-number>` for phases 1–3 / 6–7 (no-op success) | Hides typos. SPEC §18.8 documents only `4` and `5`; anything else is almost certainly a caller bug. |
| Print failures to stdout | Convention across `skills/scripts/` (and the existing claim shims) is errors → stderr, data → stdout. This script produces no data on success; on failure, the offenders are diagnostic and belong on stderr. |
| Stop at first failure | Iterating to completion lets one run surface every missing prerequisite. The caller fixes them in one batch instead of `re-run, fix one, re-run, fix one`. |
| Pass `<stocks-dir>` as a CLI arg | SPEC §18.8 signature is `validate-prerequisites <phase-number>` — single positional arg. Test isolation goes through `CLAIM_CTL_ROOT`, same as the other M1 scripts. |
| Use `grep -l` / `xargs` to chain across files | Doesn't let us emit "ticker: missing pattern" diagnostics; the loop is clearer for the small file counts (≤ 50–100 tickers per market). |

## Risks and open questions

- **R1. Continuation lines for `[Fund]` not validated.** SPEC §7.3 shows `[Fund]` content can wrap onto continuation lines starting with `|` (e.g. `| ROE: …`). Header-only validation passes a stock file that has `[Fund]` but no metrics body. Acceptable: investigation §6.5/§6.6 explicitly scopes M1 to header presence; body validity is M9 (stock linter) / M10 (scoring engine).
- **R2. `## Scoring` regex tolerates trailing whitespace only, not trailing content.** Same shape as [complete-scoring.sh](../../skills/scripts/complete-scoring.sh) / [complete-persona.sh](../../skills/scripts/complete-persona.sh). Documented in investigation §6.5; do not relax without a SPEC change.
- **R3. Filename = ticker convention.** `basename "$f" .md` assumes the file stem is the ticker. SPEC §7.3 and §17 both use `{TICKER}.md` so this holds. If a future migration adds versioned filenames (e.g. `INFY.NS.v2.md`), update both this script and the claim-ctl directory layout in lockstep.
- **R4. `set -euo pipefail` interaction with `grep -q` exit 1.** Bash's `||` keeps the script alive after `grep -q` returns 1 (`if ! grep -q ...`), so `set -e` does not abort the loop. Verified mentally; tested in the smoke checklist.
- **R5. Large stocks-dir performance.** Linear scan; one `grep -qE` per (file, pattern). For India-first scope (≤ ~50 tickers per run) this is sub-second. If portfolio scope ever climbs into thousands, switch to a single `grep -lE` per pattern with set difference — out of scope for M1.
- **No open questions for the user.** All four design choices (empty-dir behavior, header-only regex, phase whitelist, exit-code split) were resolved in the session conversation that preceded this plan.
