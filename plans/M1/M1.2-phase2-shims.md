## Scope

In scope — four executable shims under [skills/scripts/](../../skills/scripts/), each ≤10 lines of real code, that source [`_claim-common.sh`](../../skills/scripts/_claim-common.sh) and call exactly one public `_claim_*` function:

| Shim | SPEC §18.8 signature | Library entrypoint |
|---|---|---|
| [`claim-stocks.sh`](../../skills/scripts/claim-stocks.sh) | `claim-stocks <batch-size> <agent-id>` | `_claim_batch "$@"` |
| [`complete-stock.sh`](../../skills/scripts/complete-stock.sh) | `complete-stock <ticker> [<ticker>...]` | `_claim_complete "$@"` |
| [`check-progress.sh`](../../skills/scripts/check-progress.sh) | `check-progress` (no args) | `_claim_check_progress` |
| [`reclaim-stale.sh`](../../skills/scripts/reclaim-stale.sh) | `reclaim-stale [<minutes>]` | `_claim_reclaim_stale "$@"` |

All four bind `CLAIM_ROOT` to `${REPO_ROOT}/temp/research/claims` (Phase 2 namespace, SPEC §17) and **do not** set `CLAIM_VALIDATOR` — Phase 2 has no completion validation; `complete-stock` just writes `.done` (investigation §3 table).

Out of scope:

- Any validator hook (Phase 4 → M1.3, Phase 5 → M1.4).
- `validate-prerequisites.sh` — separate script, doesn't source the lib (M1.5).
- Concurrent / stale / completion tests (M1.6 / M1.7 / M1.8).
- `skills/scripts/README.md` (M1.9). Per-shim usage docs live there, not in this subtask.
- `shellcheck skills/scripts/*.sh` epic-level sweep (M1.10). M1.2 runs shellcheck locally on the four new files but doesn't gate on the full sweep.
- Per-ticker dir seeding under `temp/research/claims/{TICKER}/` (M7/M8). Shims treat existing dirs as input.

## Approach

### Shim shape (one canonical pattern, repeated 4×)

Each shim is a 6–8 line bash file:

```bash
#!/usr/bin/env bash
# <shim-name>.sh — Phase 2 <one-line behavior>. SPEC §10.3, §18.8.
# Usage: <shim-name>.sh <args-per-§18.8>
set -euo pipefail
# shellcheck source=./_claim-common.sh
source "$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)/_claim-common.sh"
CLAIM_ROOT="${REPO_ROOT}/temp/research/claims"
_claim_<entrypoint> "$@"
```

Two things to note about that pattern:

1. **Source first, then set `CLAIM_ROOT`.** [`_claim-common.sh:17`](../../skills/scripts/_claim-common.sh#L17) computes `REPO_ROOT` from its own `${BASH_SOURCE[0]}` (or `$CLAIM_CTL_ROOT` test override). The shim sources first so it can compose `CLAIM_ROOT` against that resolved value rather than re-deriving the repo root itself. Test fixtures override `CLAIM_CTL_ROOT` and the shim transparently follows.
2. **`# shellcheck source=./_claim-common.sh` directive.** Required by [M1-investigation §3](../../research/milestones/M1-investigation.md#L62) and tracked in M1 epic acceptance #4 (`shellcheck skills/scripts/*.sh` clean).

### Per-shim specifics

| Shim | Function call | Notes |
|---|---|---|
| `claim-stocks.sh` | `_claim_batch "$@"` | Library validates `<batch-size>` is a positive integer ([_claim-common.sh:79](../../skills/scripts/_claim-common.sh#L79)) and `<agent-id>` is positional. Per investigation §7.5 the agent-id format (URL-safe, no `=`) is the **caller's** responsibility — shim does not validate. |
| `complete-stock.sh` | `_claim_complete "$@"` | No `CLAIM_VALIDATOR` set. Library still enforces the prerequisite-`.claimed` check, idempotent `.done`, and multi-arg partial-failure semantics ([_claim-common.sh:105](../../skills/scripts/_claim-common.sh#L105), investigation §7.2). |
| `check-progress.sh` | `_claim_check_progress` (no args) | Strict to SPEC §18.8: no positional args. Stale-count threshold uses the library default of 30 (`CLAIM_STALE_MINUTES_DEFAULT`). Forwarding `"$@"` is rejected — there's no SPEC-documented call site for `check-progress 60`. |
| `reclaim-stale.sh` | `_claim_reclaim_stale "$@"` | Forwards optional `<minutes>` arg (SPEC §18.8 + acceptance #3 explicitly tests `reclaim-stale.sh 30`). Library validates non-negative integer. |

### File mode

All four shims `chmod +x`. They're invoked directly (`skills/scripts/claim-stocks.sh 5 agent-A`) and via `bin/run-skill skills/scripts/claim-stocks.sh 5 agent-A` (Docker venue, investigation §1). The execute bit must be set on the host.

## Rejected alternatives

| Alternative | Why rejected |
|---|---|
| One dispatcher script (`claim-ctl phase2 <subcmd> ...`) inside `skills/scripts/` | Investigation §3 finalized Option B (one file per SPEC §18.8 sub-command). A dispatcher would re-introduce branching the library was built to avoid. |
| Have shims compute their own `REPO_ROOT` independently of the library | Two sources of truth for repo-root resolution, two breakage modes for the `CLAIM_CTL_ROOT` test override. The library already exposes `REPO_ROOT` post-source; just use it. |
| `check-progress.sh "$@"` (forward args to library) | SPEC §18.8 documents no args. Adding an undocumented arg path leaks an internal library knob into the public surface for no caller. |
| Inline `set -o noclobber` write inside each shim | The atomic primitive is library territory (M1.1). Duplicating it here breaks the "thin shim" contract from investigation §3. |
| Validate `<agent-id>` format inside shim | Investigation §7.5 explicitly leaves this to the caller. Validation here would diverge from the Phase 4/5 shims (M1.3/M1.4) and bloat the shim. |

## Risks and open questions

- **R1. `set -euo pipefail` from shim conflicts with library default.** Both the shim header and [`_claim-common.sh:12`](../../skills/scripts/_claim-common.sh#L12) set `pipefail`. Re-enabling identical options is a no-op; calling this out only because investigation §6.8 lists `set -euo pipefail` as a style requirement — both files keep it independently so each is self-contained when read in isolation.
- **R2. `bin/run-skill` execution-bit propagation through Docker bind-mount.** The execute bit on the four shims is host-side; VirtioFS preserves mode bits, so this is not a concern. Investigation §1 confirms VirtioFS forwards file ops faithfully. Smoke this in M1.10 if needed; not in scope here.
- **R3. `BASH_SOURCE[0]` vs `$0` resolution under unusual launchers (e.g. `bash skills/scripts/claim-stocks.sh ...`).** Both work for an executed shim. Using `${BASH_SOURCE[0]}` matches the library's own pattern at [`_claim-common.sh:17`](../../skills/scripts/_claim-common.sh#L17) for consistency.
- **No open questions for the user.** The one design choice that could have been a question — `check-progress` arg passthrough — is decided in this plan (strict to §18.8: no args). Trivially reversible if a caller appears later.
