## 2. Approach

### 2.1 `_handle_full` composition

```python
def _handle_full(args):
    philosophy = my_philosophy.load_philosophy(args.philosophy)
    metrics    = my_philosophy.load_metrics(args.metrics)
    if metrics.get("ticker") != args.ticker:
        _die_bad_input("ticker_mismatch", f"--ticker {args.ticker!r} != metrics.ticker {metrics.get('ticker')!r}")
    thresholds = my_philosophy.check_thresholds(philosophy, metrics, args.scheme, args.sector_exception)
    persona    = my_philosophy.persona_my_philosophy(philosophy, metrics, args.scheme, args.sector_exception)
    _emit({"thresholds": thresholds, "my_philosophy": persona})
```

Both inner functions are idempotent and already tested in M3.3. `full` adds no new math — pure composition.

### 2.2 Output shape (investigation §7)

```jsonc
{
  "thresholds":     { /* check-thresholds output — §9.3 pass/fail + PF */ },
  "my_philosophy":  { /* persona --persona my-philosophy — sub_scores/signal/confidence */ }
}
```

Ticker is already present inside each nested payload. No top-level duplication.

### 2.3 SPEC §18.3 amendment

Current:
```
scoring-engine full \
  --ticker <T> \
  --metrics <metrics.json> \
  --philosophy <philosophy.md>
  → combined output (thresholds + my-philosophy persona)
```

Amended (additive):
```
scoring-engine full \
  --ticker <T> \
  --metrics <metrics.json> \
  --philosophy <philosophy.md> \
  --scheme {non_financial|banking_nbfc} \
  [--sector-exception <name>]
  → combined output {thresholds, my_philosophy}
```

Rationale: `check-thresholds` requires `--scheme`; `full` dispatches to it. Phase 4 prompt (M10) already detects scheme and exception per investigation §2 (Q3); passing them through is consistent. The amendment is additive — no prior callers broken (there are none outside M3 tests).

### 2.4 Determinism

`full` output is byte-equal on byte-equal input because both composed payloads are. No new sources of nondeterminism introduced.

### 2.5 Ticker-mismatch guard

`--ticker` is useful as a "what-you-asked-for" assertion: the Phase 4 prompt already knows the ticker from `portfolio-snapshot.csv` and passes it in; a silent mismatch with `metrics.ticker` would hide a prompt bug. Exit-1 on mismatch, structured JSON on stderr.

### 2.6 Fundamentals-first invariant test (SPEC §9.2)

Two test bodies, one file:

**Engine-side test** (`test_fundamentals_first_my_philosophy_pf`):
- Run `scoring-engine full --metrics tests/scoring-engine/fundamentals_first.json --philosophy input/india/philosophy.md --scheme non_financial --ticker SYNTHCO`.
- Assert `thresholds.philosophy_fit_graded == 15` (fixture is designed all-pass).
- Assert `my_philosophy.weighted_score == 15`.
- Asserts the fixture still reflects its M3.3 design intent and `full` passes values through unchanged.

**Spec-consistency invariant test** (`test_fundamentals_first_invariant_spec_math`):
- Hardcode SPEC §9.1 max sub-scores: `F=35, BS=20, V=20, PF=15, N=10` (sum=100).
- Hardcode the §9.2 example point: `worst_pf = 0, worst_news = 0`.
- Compute `worst_case = F + BS + V + worst_pf + worst_news = 75`.
- Assert `worst_case >= 60` (Conviction Hold floor per §9.1 bands).
- Regression guard: if a future milestone rebalances max sub-scores (e.g. F=40, BS=15), this test flags the invariant break immediately.

The second test is a **constants test** — doesn't invoke the engine. It exists because the invariant is a math property of the rubric, not a property the engine computes. Leaving it out would mean a SPEC edit could silently break the invariant.
